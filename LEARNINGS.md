### OffscreenCanvas + SharedArrayBuffer worker scaffold (2026-04-22 — Phases 0-7)

- **Issue:** Multiple long-standing main-thread bottlenecks: (1) preload wall-clock blocked by main-thread scene construction (~600-1200ms of LPCA + heightfield + post-processing on cold load); (2) in-game frame jank from the renderer competing with `mountApp.frame()` game systems on a single thread; (3) title-screen smoothness compromised by the dock-preview rAF loop sharing the thread with title flow boot work. Plus the structural problem that `CharacterScenePreview` is a 7,111-LOC monolith that would have to be touched whenever any of these were addressed.

- **Cause:** Architectural — the entire Three.js stack (renderer + scene + post-processing + per-frame loop) lives on the main thread alongside everything else (DOM, audio, input, store mutations, save persistence). Single-threaded by design since the project's inception. No off-main-thread offload existed; the only main-thread relief came from `requestIdleCallback` slicing and dynamic-import chunk splits, which both have hard ceilings.

- **Solution:** Big-bang migration of the renderer + scene to a Web Worker via `OffscreenCanvas`, with `SharedArrayBuffer` (gated by COOP/COEP `credentialless` headers) backing zero-lag main-thread polling of worker-produced state. Worker owns: `WebGLRenderer`, `THREE.Scene`, `THREE.PerspectiveCamera`, post-processing, rAF loop. Main owns: DOM, audio, input listeners, GameStore + `localStorage` save, lobby (CSS2D nametags). 60+ polled getters (avatar position, camera state, gather progress) move from synchronous JS calls to `Atomics.load` reads on a 1280-byte SAB — ZERO postMessage round-trips on the hot path. Sparse event-shaped traffic (audio SFX, gather completion, custom window events) routes via typed `MainMessage` postMessage; everything per-frame uses SAB. Capability gate (`probeWorkerCapabilities()` in `src/main.ts`) detects `OffscreenCanvas + transferControlToOffscreen + SharedArrayBuffer + crossOriginIsolated` at boot; browsers without all four fall through to legacy main-thread path until Phase 3.x reaches parity, then unsupported browsers will see an "update" message.

- **What shipped this session (Phases 0-7 scaffolding):** complete worker module skeleton (`src/worker/{AGENTS.md, protocol.ts, sharedState.ts, capabilityGate.ts, renderWorker.ts, characterSceneController.ts, workerBridge.ts}`), main-side host (`src/visual/characterSceneHost.ts`), input forwarder (`src/world/workerInputForwarder.ts`), audio router (`src/audio/workerAudioRouter.ts`), COOP/COEP headers in `netlify.toml` + `vite.config.ts`, capability probe wired into `src/main.ts`, full architecture doc (`docs/WORKER_ARCHITECTURE.md`), session report (`docs/SESSION_2026_04_22_OFFSCREEN_CANVAS_SCAFFOLD.md`), AGENT_CONTEXT update with module split + carve-outs, lobby (`multiplayerAvatarStage.ts`) annotated as permanent main-thread carve-out. Module-portability audit confirmed all scene-construction modules (forest, night magic, harvest nodes, mushrooms, dock environment, heightfield, LPCA builders, idbCache) are worker-safe; `graphicsTier.ts` and `persistentCache.ts` stay main-only by design.

- **What's deferred to Phase 3.x (next session):** the full controller build-out — calling `createRendererAsync` against the OffscreenCanvas, running scene attach (forest + character rig + equipment + harvest + decor + post-processing) inside the worker, writing real per-frame values to all SharedRenderState slots, and finally wiring `dockPreload.ts` to construct `CharacterSceneHost` instead of `CharacterScenePreview` when the gate passes. Estimated 3-4 weeks of focused work per the original plan. The scaffold is structured so each migration step ships independently behind `?worker=1` URL flag without re-establishing the bridge or breaking the legacy default path.

- **Why COOP/COEP `credentialless` instead of `require-corp`:** `require-corp` would block any cross-origin resource that doesn't send `Cross-Origin-Resource-Policy: cross-origin`. The page loads fonts from `fonts.googleapis.com` + `fonts.gstatic.com` and the vibejam widget from `vibej.am` — none of which send CORP headers. `credentialless` allows cross-origin no-credentials resources without the third-party origins needing to cooperate; it still gates `SharedArrayBuffer` allocation. Browser support: Chrome 96+, Edge 96+, Firefox 110+, Safari 17.4+. Older browsers fail the capability gate.

- **Why SharedArrayBuffer instead of postMessage state mirroring:** `CharacterScenePreview` exposes 60+ getters polled per-frame from main (HUD projection, damage floaters, magical reticle, gather state). postMessage round-trip would queue state on the receive side with at minimum 1 frame of lag — visible as input/aim sluggishness during combat. SAB lets main read worker-written state via `Atomics.load` synchronously; no IPC, no measurable latency. `Atomics.wait` is forbidden on main (would freeze UI), but polling is what the game loop already does. Atomics on `Int32Array` provide torn-read protection; `Float32Array` writes are atomic at the 4-byte word level on all major runtimes (single-writer per slot in our model, so plain assignment is correct + faster than `Atomics.store`-via-bitcast). The SAB is 1280 bytes total (256 f32 + 64 i32 slots) — trivial allocation.

- **Why scaffold-now-flesh-later instead of one big complete migration:** `CharacterScenePreview` is 7,111 LOC. Rewriting it inside the worker would be a multi-session effort regardless of strategy. Shipping the scaffold now means: (1) the message protocol + bridge + host + controller compile + typecheck end-to-end against TypeScript's exhaustiveness checks — no API drift between sessions; (2) the COOP/COEP headers + capability gate land + are smoke-testable today; (3) future sessions can populate ONE controller method per pass (e.g. wire `applyCharacterPreset` end-to-end against a real renderer), each visibly improving worker-mode parity, without re-establishing infrastructure. The legacy path keeps shipping unchanged behind the gate so users see no regression during the multi-week buildout.

- **Module split (canonical, do not undo):** Worker-side: scene + renderer + camera + post-processing + rAF + LPCA construction + heightfield + idbCache. Main-side: DOM (canvas owner, overlays, presence HUD), audio (`AudioContext`), input listeners, `GameStore` + `localStorage` save, service worker registration, lobby `multiplayerAvatarStage.ts` (CSS2D nametags require DOM — permanent carve-out, has its own secondary `WebGLRenderer`), `graphicsTier.ts` (uses `window.matchMedia` + `localStorage`; resolved tier passed to worker via `init.graphicsTier`), `persistentCache.ts`. The split is enforced by `src/worker/AGENTS.md` (no DOM, no Web Audio, no localStorage, no GameStore in worker code).

- **Files (added):** `src/worker/AGENTS.md`, `src/worker/protocol.ts`, `src/worker/sharedState.ts`, `src/worker/capabilityGate.ts`, `src/worker/renderWorker.ts`, `src/worker/characterSceneController.ts`, `src/worker/workerBridge.ts`, `src/visual/characterSceneHost.ts`, `src/audio/workerAudioRouter.ts`, `src/world/workerInputForwarder.ts`, `docs/WORKER_ARCHITECTURE.md`, `docs/SESSION_2026_04_22_OFFSCREEN_CANVAS_SCAFFOLD.md`.

- **Files (modified):** `netlify.toml` (COOP/COEP `credentialless` on `/` + `/index.html`), `vite.config.ts` (dev-server headers + `worker: { format: 'es' }`), `src/main.ts` (capability probe + Atomics smoke test at boot), `src/visual/multiplayerAvatarStage.ts` (added "stays on main thread, intentionally" comment block), `AGENT_CONTEXT.md` (added §8.1 OffscreenCanvas worker architecture section with module-split table).

- **Generalisable:** **(1) Big monolithic refactors should ship the scaffold first, content second.** The scaffold (folder structure, type-safe message protocol, bridge pattern, capability gate, doc) is INFRASTRUCTURE — it should land in one session as a coherent unit. The content (per-method migration of the 7,111-LOC monolith) is incremental and ships behind a flag. The two never block each other. **(2) When migrating across a thread boundary, identify polled-getter call sites first — they're the architectural constraint that drives everything else.** A method called per-frame across the boundary CANNOT use postMessage; it must use SAB or a pre-mirror with explicit staleness. Counting these (60+ in our case) is what made it obvious the migration needed `SharedArrayBuffer`, not just `OffscreenCanvas`. **(3) `Cross-Origin-Embedder-Policy: credentialless` is the right default for apps that load any cross-origin no-credentials resources** (Google Fonts, third-party widgets, CDN scripts). `require-corp` is stricter but breaks the long tail of well-behaved cross-origin loads. Both unlock SharedArrayBuffer the same way. **(4) Permanent carve-outs are a feature, not a defeat.** CSS2DRenderer cannot move to a worker — accept the dual-renderer model and document it loudly. The lobby's secondary `WebGLRenderer` costs essentially nothing (lobby isn't perf-critical) and the alternative (replacing CSS2D with `THREE.Sprite` text labels) noticeably degrades nametag crispness. **(5) Capability gate before behavior change.** Probe + log capability at boot in a separate file; let the migration FLIP a single switch from "log" to "block" once parity ships. Keeps the capability check and the policy decision in different commits, different sessions, and different review cycles.

- **Do NOT do:** (1) Add `WebWorker` to the global `tsconfig.json` `lib` array — it conflicts with `DOM` on `self`, `postMessage`, and other shared globals. Use a local minimal interface declaration in the worker file (`interface WorkerSelf { postMessage(...) ... }; const workerSelf = self as unknown as WorkerSelf;`) so the worker file is self-contained without forking the whole-app tsconfig. (2) Use `postMessage` for state that's read per-frame — it adds 1 frame of lag and ~50-200μs IPC cost per round-trip. Reserve postMessage for sparse event traffic; use SAB for everything per-frame. (3) Put the `WebGLRenderer` construction in the worker before verifying the engine's transitive imports stay worker-portable. The empire-engine bootstrap may pull in `window`-touching code that needs to be fenced off first. Phase 4 audit covers the scene modules but Phase 3.x will need to verify the engine bootstrap separately. (4) Forget the `transferControlToOffscreen` MUST be in the postMessage transfer list — without it the canvas is structuredCloned, which throws (canvases aren't cloneable). (5) Try to call `Atomics.wait` from the main thread to sync with the worker — main-thread `Atomics.wait` throws synchronously by browser policy. Polling reads (which the game loop already does) are sufficient.

### `bush_collapse` degenerate-matrix screen flash (2026-04-22 — Phase G)

- **Issue:** Player report 2026-04-22: "the whole screen flashes when I harvest berries during the phase where the bush disappears." Initial diagnosis was wrong (assumed bloom on emissive berries). Player follow-ups narrowed it down: not just berries, also vines + heather + purple ground covers; not bloom; awakened mode only; happens during the disappear phase of the break animation (not at the moment of break).

- **Cause:** Long-standing geometry bug in `bush_collapse` animation, exposed by the awakened-only magic-projectile harvest path. Both `freeRoamHarvestNodes.ts` and `dockForestBatchedScene.ts` had identical formulas:
  ```ts
  const sh = Math.max(0, 1 - t);            // horizontal shrinks linearly
  const sv = Math.max(0, 1 - t * 1.4);      // vertical "collapses faster"
  ```
  The asymmetric rates caused `sv` to hit 0 at `t = 0.714` while `sh` still had ~28 % to shrink. For that final 28 % of the animation the scale matrix was `(sh, 0, sh)` — a **singular matrix with a zero column**. Three.js renders the InstancedMesh with NaN/inverted normals (no Y component for the cross product); the shadow pass projects a degenerate caster from the directional light's view, producing moiré / artifacts across the entire shadow map; the artifacts modulate ALL shadowed surfaces at once → reads as a screen-wide flash. Awakened-only because magic projectiles are the only way to break bushes; dream mode never triggers the animation. The bug presumably existed since the animation was first written (initial commit per `git log --follow`) but only surfaced now because the recent shadow softening (Phase D) and shadow-tier flips (Phase E6.6) made the shadow pass more sensitive to degenerate casters.

- **Solution:** Change vertical-scale formula from `sv = max(0, 1 - t * 1.4)` to `sv = sf * sf` where `sf = max(0, 1 - t)`. `sf²` shrinks faster than `sf` (preserving the "flatten faster than shrink" feel — at t = 0.5: sh = 0.5, sv = 0.25, already half-flat) but reaches 0 at the SAME frame as `sh` (t = 1.0). The matrix is never singular at any intermediate frame — every column has a nonzero scale until the animation ends and the instance becomes invisible regardless. Applied identically to both `freeRoamHarvestNodes.ts` and `dockForestBatchedScene.ts` since both share the same animation contract.

- **Why this respects existing LEARNINGS gotchas:** (1) **Animation visual idiom preserved** — `sv = sf²` still squashes the bush flatter than it shrinks horizontally (the "GoE bush_collapse" character). At t = 0.5 the bush is at 50 % horizontal × 25 % vertical → already visibly flatter than it is wide, which reads as the same "collapse" visual as the original. (2) **No per-instance state added** — pure formula change, no new fields, no pool reshape. (3) **Both harvest systems patched** — the bug existed in both because the animation contract is shared; fix lands identically so future contract evolutions don't drift. (4) **No save-state migration** — pure runtime visual change.

- **Result:** Screen no longer flashes when bushes / berries / vines / heather are harvested in awakened mode. Animation visual is essentially identical (the squared falloff makes the squash slightly more dramatic in the middle but the same time-to-disappear). `npx tsc --noEmit` exit 0. No lint errors.

- **Files (modified):** [src/world/freeRoamHarvestNodes.ts](src/world/freeRoamHarvestNodes.ts) (`bush_collapse` formula `sv = max(0, 1 - t * 1.4)` → `sv = sf²`), [src/world/dockForestBatchedScene.ts](src/world/dockForestBatchedScene.ts) (same change for the batched dock-forest harvest path).

- **Generalisable:** **Asymmetric scale-down animations CANNOT have any axis reach zero before the others.** Three.js (and most 3D renderers) interprets a matrix with a zero column as singular — the cross product for normal recovery returns zero, lighting degenerates to ambient-only, shadow casters project as degenerate planes, and depth-buffer values become unstable. Always couple the falloffs so all axes reach zero on the same frame: use `sv = sh × f(t)` where `f(t) > 0` until the very end, or use `sv = sh^k` for `k > 1` (faster collapse) or `k < 1` (slower collapse). **For a "flatten faster than shrink" animation, `sv = sh²` is the canonical pattern** — preserves the squash visual without ever producing a degenerate matrix. **When a visual bug surfaces only in awakened mode, suspect interaction with awakened-only systems (mobs, magic projectiles, damage floaters, lock-on)** — the awakened systems often expose code paths that dream mode never triggers. In this case, magic projectiles are the only way to break bushes, so the bug had been latent in the animation code for the entire dream-mode lifespan.

- **Do NOT do:** (1) Couple asymmetric animation rates with `Math.max(0, ...)` clamps without thinking through the singular-matrix case. The `Math.max` only protects against negative scales — it does NOT prevent the "one axis hits zero before the others" problem, which is the actual hazard. (2) Trust that "the geometry just becomes invisible at scale 0" — Three.js still processes degenerate scaled instances through the shadow pass, the depth pass, and any custom shaders. The visual bug surfaces in those secondary passes, not the main render. (3) Suspect bloom or post-processing for "screen flashes" before checking animation matrices in the shadow caster path. The diagnostic order should be: (a) check if the flash correlates with a specific animation in the scene; (b) if yes, check if any instance scale or matrix is degenerate during that animation; (c) only THEN look at post-processing as a contributor. Spent ~10 minutes hunting bloom/emissive theories before the player's "vines and heather too" clarification revealed the bush_collapse common factor. (4) Apply a quick fix to one harvest system without auditing the other. Both `freeRoamHarvestNodes` and `dockForestBatchedScene` had the identical animation contract — fixing only one would have left the bug present in the other path. The dual-fix is the right unit of change.

### Tier hierarchy reshape — cheap bloom on `'perf'`, decoupled buffer-size from quality tier, shadow softening reverted (2026-04-22 — Phase F)

- **Issue:** Phase E (earlier same day) shipped a "sharpness recovery" pass that included three coupled changes — device-aware default `'balanced'` on high-tier, DPR cap loosened on `'balanced'` 1.0 → 1.5, pixel budget bumped 3.11 MP → 4.10 MP — plus shadow softening (radius 2.8 → 5.5, bias -0.00012 → -0.0005, normalBias 0.02 → 0.045) and re-enabled far-tree shadows on high tier. Player report 2026-04-22 follow-up: **(a)** "shadows look like voxel blocks ... like minecraft version" — the shadow softening + far-tree-shadow re-enable produced visible PCF banding at shadow boundaries; **(b)** "we don't need" the apple-branch fix — silhouette reads cleanly enough without visible branches; **(c)** "default needs to be perf and set better so scene is enhanced but we don't lose fps. Balanced drops us to 30-40 fps and that is too low but looks amazing — can we get balanced mode quality but high fps?" The Phase E `'balanced'` config was 60 FPS → 30-40 FPS on the player's machine, even though they liked the look.

- **Cause:** Three independent diagnoses, each illustrating a "the wrong axis was used to differentiate quality tiers" pattern:
  - **Shadow voxel banding from stacked Phase D + Phase E changes.** Shadow map was 2048 × 2048 covering ±14 world units (texel size = 14 mm world). PCF kernel radius bumped to 5.5 texels = 77 mm world filter spread. Phase E6.6 then added far-tree casters into the same shadow map → MORE shadow geometry competing for the same texel budget → texel density at the edge of the frustum dropped just as the soft PCF kernel's filter span grew. Result: the kernel wrapped across visibly-quantized texel boundaries on edge-of-frustum trees → "voxel blocks" reading. Each individual change was reasonable in isolation; the regression came from the COMBINATION.
  - **`'balanced'` tier was distinguished from `'perf'` by the wrong axis.** Phase E6 made the gap "DPR 1.0 + 2.07 MP buffer + bloom on" (`'balanced'`) vs "DPR 1.5 + 4.10 MP buffer + bloom on" (`'perf'`)... wait, that's backwards — Phase E6 actually upgraded `'balanced'` to DPR 1.5 + 4.10 MP. The gap was: "DPR 1.0 + 2.07 MP buffer + cheap bloom" vs "DPR 1.5 + 4.10 MP buffer + full bloom." That's **2.25× fragment work × 1.32× pixel budget = ~3× more fragment cost** for `'balanced'` vs `'perf'`. Bloom is ~3-5 ms; the buffer/DPR jump dwarfed the bloom cost. So `'balanced'` was paying 60-80% MORE fragment cost than `'perf'` for an upgrade that the player could mostly attribute to bloom (the visually-dominant feature). The buffer-size axis was eating the FPS budget without contributing proportionally to perceived quality.
  - **The apple-branch fix solved the wrong problem.** Phase E1 carefully fixed the branch orientation (rotation order YZX + base π/2 + position offset), but the player's preference was actually "no branches at all" — the apple silhouette reads cleanly from the crown sphere + accents + apples without needing branches to communicate "this is a tree." Engineering-wise the fix was correct; product-wise the feature wasn't wanted.

- **Solution:** Three subtractive fixes:
  - **(F1) Revert shadow softening to round-3 values.** [src/visual/characterScenePreview.ts](src/visual/characterScenePreview.ts) — `shadow.bias` `-0.0005 → -0.00012`, `shadow.radius` `5.5 → 2.8`, `shadow.normalBias` `0.045 → 0.02` (all back to round-3 baseline that had been working since 2026-04-19 lighting overhaul without complaint). [src/visual/forestEnvironment.ts](src/visual/forestEnvironment.ts) — `mergedForestFar.castShadow` reverted to unconditional `false`. Far-tree shadows can return in a future round but only with either a dedicated cascade or a 4096+ shadow map so texel density doesn't drop at distance. Kept Phase D's `useBasicShadowMap: false` on low tier — `PCFSoftShadowMap` is universally better than `BasicShadowMap` and that change wasn't part of the regression.
  - **(F2) Apple branches removed entirely.** [src/visual/goeStyleHarvestLPCA.ts](src/visual/goeStyleHarvestLPCA.ts) — branch loop deleted, `barkLight` material deleted (unused now). Fewer draw calls per apple tree, no bug surface area.
  - **(F3) Tier hierarchy reshape: decouple bloom/SSAO from buffer size.** [src/visual/characterScenePreview.ts](src/visual/characterScenePreview.ts) `computeEffectivePixelRatio` — `'perf'` AND `'balanced'` now share the same buffer size (DPR cap 1.0 + pixel budget 2.07 MP); only `'full'` uncaps. [src/visual/characterScenePreview.ts](src/visual/characterScenePreview.ts) `applyDockPostProcessing` — `'perf'` enables CHEAP bloom (`bloomStrength: 0.18 / bloomRadius: 0.18 / bloomThreshold: 0.94`); `'balanced'` adds CHEAP SSAO (`ssaoResolutionScale: 0.35 / ssaoKernelSize: 8`) on top of full-strength project-default bloom. Net visual gap between tiers is now post-processing intensity (bloom strength + SSAO presence), NOT buffer resolution. Cost gap shrinks dramatically: `'perf'` ≈ baseline + 3-5 ms bloom; `'balanced'` ≈ baseline + 5-8 ms (full bloom + cheap SSAO); `'full'` uncapped for power users. (2) **`awakenedFullQuality = '1'` legacy opt-in still wins** — power users who explicitly chose `'full'` keep it. (3) **No save-state migration** — all changes are runtime tier defaults + visual geometry. (4) **Round-3 shadow values verified-working** — those were the values shipped 2026-04-19 → 2026-04-22 morning without any shadow complaints. Reverting to a known-good config is safer than incremental retuning. (5) **Bloom + SSAO project defaults preserved** — `project.json` still has `bloom: true / ssao: true` so opt-up tiers can read full-quality settings; tier overrides only WEAKEN them, never override true → false except for SSAO on `'perf'`. (6) **Per-frame work unchanged** — bloom + SSAO are post-processing passes, not per-frame scene logic; no impact on the systems audited in Phase E3.

- **Result:** Default tier `'perf'` now ships with cheap bloom enabled — magic projectiles, mushroom caps, lanterns, and emissive props all glow softly without paying the buffer-size cost. - **Generalisable:** **When designing quality tiers, decouple "buffer size" from "post-processing intensity" — they're independent dials and most players want one without the other.** Bumping DPR/pixel budget on `'balanced'` was a 2-3× fragment cost increase that the player attributed almost entirely to "bloom looks great" — meaning the buffer-size axis was wasting FPS for a feature the player wasn't actually getting visual credit for. Better tier design: pick ONE axis to differentiate per tier hop. `'perf' → 'balanced'`: same buffer, more post-processing. `'balanced' → 'full'`: more buffer, max post-processing. Each tier hop is meaningful AND the cost gap matches the visual gap. **For "minecraft-block shadows", the cause is almost always one of: (a) `BasicShadowMap` shadow type, (b) PCF kernel radius too high relative to shadow map texel density, (c) shadow map resolution too low for the camera frustum bounds, or (d) too many casters competing for shadow texels.** The diagnostic flow: check `renderer.shadowMap.type` first; if `PCFSoftShadowMap`, check `shadow.radius` against `mapSize / camera_frustum_world_units`; if texel size × radius > ~50mm world, expect visible banding on edge-of-frustum casters. **Verify-by-revert beats incremental retuning for visual regressions.** Phase D shadow values (radius 5.5 / bias -0.0005 / normalBias 0.045) were chosen by reasoning ("softer is better, looser bias avoids peter-panning"). The revert to round-3 (2.8 / -0.00012 / 0.02) was chosen by track record (those values shipped for 3 days without complaint). Project.json's `ssaoResolutionScale: 1.0` default is too aggressive for any tier short of `'full'` — tier overrides should drop it to 0.35-0.5 unless the player explicitly opts up. **Apple trees (and most LPCA broadleaf trees) read cleanly from "trunk + crown sphere + accents + fruit" alone** — branches are an aesthetic choice, not a silhouette necessity. Removing them when they're broken is often the right call vs trying to re-engineer their orientation; the silhouette doesn't suffer.

- **Do NOT do:** (1) Differentiate two adjacent quality tiers by ANY combination of axes that requires "the player expects a 2-3× FPS drop." Every tier hop should produce a visible visual difference AND a proportional cost difference — if the cost gap is much bigger than the visual gap (Phase E6's `'balanced'` was ~3× cost for ~30% perceived visual upgrade), the tier picker becomes useless because no player will want the middle option. (2) Change multiple shadow parameters simultaneously and ship without verifying each change in isolation. Phase D's three shadow tuning changes (bias + radius + normalBias) ALL went in together — when the regression surfaced, there was no way to know which one was the actual culprit. The right rollout for "soften the shadows": change ONE param, verify in scene, change next, verify, etc. (3) Add far-tree shadows (or any new shadow casters) without first verifying the shadow camera frustum has room for them at the target texel density. The Phase E6.6 `mergedForestFar.castShadow = graphics.tier === 'high'` change was correct in isolation (high-tier GPUs CAN afford the extra rasterization) but combined with Phase D's wider PCF kernel it pushed the shadow map past its useful resolution. Either bump shadow map size FIRST when adding casters, or use a separate cascade. (4) Accept "looks amazing but 30 FPS" as a tier configuration. Even if power users explicitly opt into the high-quality tier, defaulting them to a config that drops to 30 FPS on their dev machine produces user reports like the one this session addressed. The right floor for a quality tier is "60 FPS on the dev's reference machine"; if the machine can't sustain that, the tier needs to be cheaper. (5) Re-engineer a visual feature that the player explicitly says they don't want. Phase E1's apple-branch fix was carefully designed (rotation order YZX + base π/2 + position offset along trunk-edge) and worked correctly — but the player's response was "we don't need them." The right move was to remove rather than to keep refining the fix. Spend engineering effort on what's wanted, not on what's debugged. (6) Trust `ssaoResolutionScale: 1.0` from a project config without challenging it. Project.json's SSAO config was inherited from earlier development and never re-evaluated against the current FPS targets — full-res SSAO is a luxury setting, not a default. When you import any post-processing config, audit each numerical parameter for "is this still appropriate at our current visual + perf targets" before shipping.

### Trippy terrain palette recalibration + shadow softening (2026-04-22 — Phase D)

- **Issue:** User reported *"shadows look horrible"* + *"our terrain didn't come out like it look in stick man project ... too much green ... the stickman has a good combination of colors in the right ratio."* Two latent visual bugs from the round-3 trippy-palette port (`docs/SESSION_2026_04_20_PRELOAD_ROUND_3.md` Phase 8l) finally surfaced when other distractions (cutscenes hiding the title, dim shadow tuning) were removed in earlier 2026-04-22 phases. The terrain rendered as a uniform mono-cyan field reading as green — none of the magenta dirt or amber rock variation the palette was supposed to provide ever appeared. Shadows on the dock + characters showed visible peter-panning (shadows detached from feet, looked floating) and step-aliased edges; on low-tier devices shadows were also blocky pixelated (no filtering at all).

- **Cause:** Three orthogonal "ported verbatim, never recalibrated for our scale" patterns:
  - **Slope thresholds tuned for stick man's terrain scale.** `bakeTrippyGroundVertexColors` was ported from `C:\stick man`'s `TerrainBuilder.GROUND_COLORS_TRIPPY` slope picker. Stick man uses `terrain.heightScale = 5`; we use `terrain.heightScale = 0.88` ([project.json:53](project.json#L53)). `computeSlopeAt` returns `sqrt((dh/dx)² + (dh/dz)²)` which is **linearly proportional to heightScale** — so a stick-man slope of `0.7` corresponds to `0.7 × (0.88/5) ≈ 0.123` here. The original thresholds (`> 0.7`, `> 0.4`, `> 0.28`, `> 0.15`) almost never triggered on our heightfield (max realistic slope ~0.05) — 99% of vertices fell into the grass band. The function's own doc-comment flagged the issue as a known limitation (*"Idle craft's awakened terrain has gentler heights than stick man ... most of the map ends up in the grass band which is the intended 'psychedelic field' read"*) but the read was wrong: "psychedelic" implies variety, not uniform cyan.
  - **6-color palette in 3 hue families is not enough for a noise-driven mix.** Even with the slope thresholds fixed, picking among only `grass / grassDark / dirt / dirtLight / rock / rockLight` produces a 3-hue field. Stick man's "good combination of colors in the right ratio" depends on having more palette entries to draw from so noise rotation produces visible variety rather than just two cyan variants alternating.
  - **Shadow tuning tuned for "sharp shadows on small props," not "outdoor terrain with characters and trees."** `shadow.bias = -0.00012` is a tight value appropriate for indoor/UI scenes with thin geometry; on outdoor LPCA characters the contact-shadow detached visibly from the feet (peter-panning). `shadow.radius = 2.8` produced visible step-aliasing at the shadow boundary. `shadow.normalBias = 0.02` was on the low side. And low tier used `THREE.BasicShadowMap` which is single-sample (no filter) → blocky pixelated edges that look broken on any GPU made since 2010 even though the perf cost of `PCFSoftShadowMap` is negligible (4-tap filter).

- **Solution:** Subtractive fixes; no new state, no new helpers beyond a 5-line `colorNoise2D` hash:
  - **(D1.a) Slope threshold scaling.** Multiply each threshold by `heightScale / 5` (`slopeRef`) so stick-man-tuned values translate to the right proportion of our gradient range. New thresholds at `heightScale = 0.88`: rock-heavy `0.123`, rock-light `0.070`, dirt-heavy `0.049`, dirt-light `0.026`. - **Generalisable:** **When porting numerical tunings between projects with different scale parameters, multiply by the scale ratio at the call site** — don't trust that "we'll see if it works." `slopeRef = heightScale / 5` here is a one-line guard against the entire dead-code class. The same pattern applies to physics constants (gravity scale), camera clipping planes (near/far ratios), and any threshold derived from a derivative quantity. **Single-hue palette + slope picker = mono-color field.** For a "psychedelic" or "varied" read, you need at minimum 8-10 palette entries across 3-4 hue families AND a noise-driven mix on top of the slope/height picker. The slope/height picker alone gives you N regions, each uniformly colored within itself; the noise mix breaks the regions internally so the player sees variety even on a flat plain. **Cheap 2D noise is fine for visual variety** — `sin()`-based hashes are not cryptographically random but are perfectly serviceable for "pick from a small palette per vertex." Don't reach for a full Perlin/Simplex unless the application needs continuity across noise space. **`THREE.BasicShadowMap` is never the right answer in 2026.** Even on the lowest device tier, `PCFSoftShadowMap` with a 1024 shadow map looks better than `BasicShadowMap` with any size. The 4-tap filter cost is negligible on every consumer GPU; the visual difference is "blocky stair-step pixels" vs "smooth penumbra." For shadow tuning: **`bias` controls peter-panning vs acne; `normalBias` controls acne specifically; `radius` controls edge softness.** The combination `bias = -0.0005 / normalBias = 0.045 / radius = 5.5` is a good outdoor-LPCA default; tighten or loosen each individually based on the specific symptom.

- **Do NOT do:** (1) Port slope-based color or physics tunings between projects without scaling by the ratio of whatever scale parameter (heightScale, gravity, character height, etc.) drives the units of the input. The original code that added a `// most of the map ends up in the grass band which is the intended 'psychedelic field' read` comment was wrong — the actual visual was "uniform green," not "psychedelic." If you find yourself writing a doc-comment that explains away an unintended behavior, fix the behavior instead. (2) Trust a 6-color palette to produce visible variety across thousands of vertices. The eye reads "6 colors in 3 families" as "3 colors with variants" — not "varied palette." Aim for 8-10 entries across 4+ families if you want noise mixing to read as variety. (3) Tune `shadow.bias` to its tightest defensible value (`-0.00012` here). The "right" bias is the loosest value that doesn't cause acne — typically `-0.0003` to `-0.001` for outdoor scenes with detailed geometry. Loosen until acne appears, then back off slightly. (4) Use `THREE.BasicShadowMap` as a "low-tier optimization." It saves nothing meaningful on any GPU made since 2010 and produces shadows that look broken to the player. `PCFSoftShadowMap` at a smaller `mapSize` is a much better budget tradeoff. (5) Treat `shadow.normalBias` as "small adjustment, can ignore." It's the ONLY parameter that fixes shadow acne without re-introducing peter-panning, and the right value for outdoor LPCA scenes is `0.04-0.05` — not the Three.js default of `0`. (6) Bake noise across a terrain WITHOUT first thinking about frequency. Too high → looks like TV static (bad). Too low → looks like one big tinted blob (also bad). Tune to "color rotates every 3-6 world units when walking across the map" — for a `0.18 cycles/unit` low-frequency layer + `0.91 cycles/unit` per-vertex speckle, a player walking 1 unit/sec sees a hue shift every ~6 seconds with constant local variation.

### Preload round 5 — frame-spread sync blocks + scheduler.postTask priorities + concurrency-limited preload imports + View Transitions extension (2026-04-21)

- **Issue:** After round 4 (Workbox SW + storage.persist + adaptive device profile + immutable headers + cutscene skip flag) the network bottleneck on judge refresh was largely gone — the bottleneck moved to **CPU work between SW response and first interactive frame**. A parallel research pass (`docs/SESSION_2026_04_21_PRELOAD_ROUND_5.md`) ranked 9 Tier-1 techniques but exploration found 5 already shipped or N/A. The remaining real targets: the four single-shot heavy sync blocks in `attachForestBackdrop` and `attachDockForestBatchedScene` (~80–300 ms tasks per inline comments) yielded BETWEEN phases but each phase was still monolithic; the 7-way `Promise.all([7 dynamic imports])` at the start of `runExtendedPreload` was a parse-storm; 4 of the major scene swaps in `main.ts` were hard cuts (only `renderPage` used View Transitions, added in round 4); 7 rIC sites were fire-once with no priority intent (and `scheduler.postTask` was unused anywhere); and the heavy forest sync work didn't expose any way to know HOW heavy it actually was without a measurement scaffold.
- **Cause:** Five orthogonal "right shape, wrong granularity" patterns:
  - **Frame-spread done at the wrong granularity.** Round 3 added `await yieldToEventLoop()` between forest-build phases (`scatterWorldForestStrata`, then `mergeByMaterialTiled` × 3, then `attachIdleCraftNightMagic`), but each phase remained a single sync block. Yielding between an 80-ms task and a 150-ms task means the browser sees two long tasks instead of one — strictly better, but neither task alone respects the 50 ms "long-task" threshold. The right granularity is **inside** the hot loop (per-iteration / per-budget-window yield), not just between phases.
  - **Parse-storm masquerading as "load in parallel."** `Promise.all([7 dynamic imports])` looks like a network-parallelism win, and on a healthy connection it is — but the resulting 7 chunk-parse tasks land on the main thread within milliseconds of each other and stack into one ~100–300 ms parse window. Total wall-clock is unchanged whether they parse pipelined-2 or stacked-7; the difference is task SHAPE, which is what blocks input + paint.
  - **Yield primitives without priority intent.** `requestIdleCallback` has one knob (`{ timeout }`) and no concept of "this is genuinely background" vs "this is what the player is waiting to see." All 7 rIC sites in the codebase used the same shape; under load the browser couldn't distinguish background SW registration from user-visible dock preload from user-blocking deferred `renderPage`. `scheduler.postTask({ priority })` (Chrome 94+, Firefox 142+) is the right API; the rIC fallback ladder preserves Safari behavior.
  - **Hard cuts where View Transitions are free.** Round 4 added View Transitions ONLY to `renderPage` (deck page navigation). The 4 high-vibe-payoff cuts — splash → curse cutscene, curse → title, `enterGame` (title → cleared appRoot), `returnToTitle` (game shell → title) — stayed as `appRoot.replaceChildren()` hard cuts. Each is a perceived-latency mask, not a real speedup, but the user feels a 200 ms cross-fade as smoother than a 50 ms hard cut.
  - **No empirical signal for "what's actually heavy."** The inline comments in `forestEnvironment.ts` said `~80–150 ms`, `~40–80 ms` × 3, etc., but those numbers were one-off measurements from round 3 with no way to re-validate after each subsequent change. ### Awaken-mode FPS round 4 — fullscreen / large-monitor total-pixel cap + aspect-preserving DPR shrink (2026-04-20)

- **Issue:** Round 3 (2026-04-18) capped DPR at 1.0 and dropped bloom + SSAO when entering awakened mode, lifting fullscreen FPS from 30-40 → 100+ on a 1080p × DPR 2 laptop. Players on 1440p / 4K / ultrawide monitors reported the **windowed** awakened mode was smooth (~80-120 FPS) but **fullscreen** still dropped into the 30-50 FPS range, even on the same 'perf' tier. The DPR cap alone wasn't enough — at 1.0 DPR a 4K canvas still rasterises 8.3 MP per frame (~4× the 1080p 2.07 MP baseline that round 3 implicitly assumed). Fragment work scaled linearly with the CSS canvas area; the windowed-mode optimisation didn't follow the canvas as it grew.
- **Cause:** `dockPreviewDrawSize` in `characterScenePreview.ts` had a hard `min(1680, w)` cap on **width only** while leaving **height uncapped**. Two compounding problems on awakened-mode fullscreen (canvas styled `100vw × 100vh`):
  1. **Aspect-ratio break.** On any screen wider than 1680 CSS pixels, the buffer was rendered at `1680 × containerHeight` while the displayed canvas was `containerWidth × containerHeight`. The browser stretched the buffer horizontally to fill the wider CSS box → the rendered scene appeared horizontally squashed (FOV felt wrong, circles read as ovals). Subtle but a real visual regression we'd shipped.
  2. **Unbounded vertical pixels.** `1680 × 2160` (4K fullscreen) = 3.6 MP, vs `1680 × 945` (windowed at 1080p) = 1.6 MP — a 2.3× fragment-work jump from the SAME line of CSS. Then on tall monitors / portrait setups it gets worse.
- **Solution:** Two-layer fix in `src/visual/characterScenePreview.ts`. Both extend the **windowed-mode optimisation** uniformly to any canvas size — the technique is now canvas-size-independent.
  1. **Drop the `1680` width cap entirely from `dockPreviewDrawSize`.** Deck-mode CSS already enforces `max-width: 1680px` on `.character-preview-root` so the JS cap was dead code there; awakened-mode is where it was actively harming. Now `dockPreviewDrawSize` returns true CSS pixel dimensions of the container — `camera.aspect = w / h` matches the displayed canvas exactly, no browser stretch.
  2. **Add a tier-aware total-pixel budget to `computeEffectivePixelRatio(cssW, cssH)`.** When `awakenedRenderBudget` is on, compute natural buffer pixels = `cssW * cssH * dpr²`. If above the tier's budget, multiply `dpr` by `sqrt(budget / natural)` — the SAME multiplier on both dimensions preserves aspect ratio. The browser does the (free) upscale to the displayed CSS box. Tier targets:
     - `'perf'`     → `1920 × 1080` ≈ 2.07 MP. Matches the round-3 implicit baseline at 1080p; auto-shrinks DPR sub-1.0 on bigger screens to hold this ceiling.
     - `'balanced'` → `1.5 × 2.07 MP` ≈ 3.11 MP. Keeps headroom for the bloom downscale chain.
     - `'full'`     → no cap (matches dream-mode behaviour for users on dedicated GPUs).
  3. **Pass `cssW`/`cssH` from constructor + `onResize`.** Both call sites already had the dims locally; just thread them through. Without them, the area cap is skipped (safe fallback).
- **Why this respects the existing LEARNINGS gotchas:** (1) `setPixelRatio` changes the renderer's framebuffer size but does NOT change shader programs — no `numDirLightShadows` / `numPointLights`-style hash flip, no first-use compile freeze. (2) The browser upscale is GPU-cheap (single triangle blit) compared to even one extra frag-shaded MP at native — net win is large. (3) FXAA stays in the post stack at all tiers so the slight downscale-then-upscale soften is masked. (4) `'full'` tier opt-out preserves the existing power-user escape hatch.
- **Result:** On a 1440p (3.69 MP) / 4K (8.29 MP) / ultrawide 3440×1440 (4.95 MP) fullscreen awakened canvas at the `'perf'` tier, expected FPS now matches the windowed FPS on the same machine — the canvas size stops being the dominant fragment-cost factor. On the test 4K HiDPI laptop (was 30-50 FPS fullscreen at `'perf'` after round 3): expect 90-130+ FPS. Aspect-ratio distortion gone. `'balanced'` tier same idea with 1.5× headroom for bloom. `'full'` tier unchanged (matches dream mode).
- **Files:** [src/visual/characterScenePreview.ts](src/visual/characterScenePreview.ts) (`dockPreviewDrawSize` width cap removed + aspect comment, `computeEffectivePixelRatio(cssW, cssH)` signature + pixel-budget block, constructor + `onResize` updated to pass dims), [src/ui/graphicsHelpCopy.ts](src/ui/graphicsHelpCopy.ts) (`'lit-awakened-quality'` help copy now mentions the total-pixel cap so the Esc menu is honest about what each tier does).
- **Generalisable:** When you're optimising for a fragment-bound scene and your "fix" relies on a fixed pixel-count assumption (DPR cap to 1.0 on a 1080p target), the optimisation is **canvas-size-coupled** — it only holds while the canvas stays at the target size. The robust shape is a **pixel-area budget** that scales the render multiplier (`renderer.setPixelRatio`) so the buffer pixel count stays bounded regardless of CSS canvas dimensions. Always preserve aspect by applying the SAME scale to both dims; never cap one dimension while leaving the other free (that breaks both aspect ratio AND the budget guarantee).

Always cap by total area. (2) Use CSS `max-width` to constrain the awakened canvas to "fix" this — it'd letterbox the play area on bigger screens, which the player perceives as lost real estate. The buffer-side downscale + browser upscale gives them the full canvas at bounded fragment cost. (3) Tie the area budget to `'fullscreen'` detection (Fullscreen API event) — the budget should apply uniformly to ANY large canvas, not just the F11 case. The current `awakenedRenderBudget` flag + area math handles every case (windowed, F11, multi-monitor swap, browser zoom change) without any extra event listeners.

### Cutscene smoothness — pre-fetch video bytes + drain JS chunk parses BEFORE playback (2026-04-19)

- **Issue:** The shattering cutscene (act 2 of the intro flow) stuttered visibly during playback even after a previous fix awaited the dock preload before the video started. Console showed `<link rel=preload> uses an unsupported 'as' value` for the first attempt at video preloading.
- **Cause:** Three compounding problems hitting the main thread during video decode:
  1. **Video file fetched on first frame.** `<video>.src = 'cutscenes/intro_the_shattering.mp4'` triggered the HTTP fetch at playback start; decoder ran while bytes were still streaming → buffer underruns → user-visible stutter.
  2. **`mountApp` chunk parse running in parallel with cutscene.** `loadMountAppModule()` was kicked non-awaited; its JS parse cost (one of the largest chunks) hit the main thread DURING video decode.

- **Generalisable pattern:** **anything that runs on the main thread during a video / animation / WebGL frame loop must be moved off the critical path.** Use `fetch(..., { cache: 'force-cache' })` to warm HTTP cache before media plays. ### 3D-aware reticle aim — terrain heightfield walk + Y-aware collision world raycast (2026-04-19)

- **Issue:** Magic projectile aim went through five iterations of "wrong direction / barely visible / blocked when shooting down". Symptoms shifted between failure modes:
  1. **Shoots at viewport** — initial raycast aim hit the player's own footprint at distance 0 (camera in 3rd-person sits behind the avatar's collision capsule).
  2. **Aim Y goes underground** — even after `ignoreOwnerId: 'player'`, the 2D raycast's `dist3d = hit.dist / xzLen` math placed the aim Y deep underground when the camera pitched steeply down.
  3. **Bolt parallel to reticle, not converging** — switched to camera-forward direction × 40 m for "always points where reticle points", but lost the convergence on close targets that the user actually wanted.
  4. **Hits the wrong tree (XZ false-positive)** — 2D raycast latched onto a tree's flat XZ footprint even when the camera pitched UP and the ray actually passed OVER the tree's canopy.
  5. **"Barely visible on screen"** — bolt fired in camera-forward direction which has y=-0.177 at neutral pitch (camera naturally tilted ~10° down to keep the avatar in frame), bolt dove into terrain in ~0.6 s.
- **Cause:** All five symptoms come from trying to derive a 3D aim point from a 2D raycast + heuristic Y reconstruction. The 2D collision world doesn't know where TALL things end vertically (a tree's footprint is a circle; the canopy's `topY` lives on the footprint but the raycast didn't read it). And derived Y positions from `forward.y * dist3d` go arbitrarily far underground at steep angles because there's no terrain stop.
- **Solution: proper 3D scene raycast.** `awakenedCombat.ts` `reticleAimPoint` now runs TWO parallel raycasts and picks the closer hit:
  1. **Terrain raycast.** Walk camera ray in 1 m steps. At each sample, check `ray.y < heightField(ray.x, ray.z)`. When the ray crosses below terrain, binary-search the bracket (6 iterations → ~0.015 m precision) to refine. Returns the 3D point where the camera ray actually hits the ground.
  2. **Y-aware obstacle raycast.** `collisionWorld.raycastXZ` extended with optional `originY` + `dirY`. Each candidate footprint's hit is filtered against the ray's Y at the hit XZ vs the footprint's `[bottomY, topY]` extent. Trees skipped when the ray passes over their canopy; mobs registered when the ray hits their chest height; under-roof shots not blocked by the roof itself.
  3. Pick the closer of the two hits in 3D distance along camera-forward. If neither hits within `maxDist`, fall back to `camera + forward × maxDist` (open-air shot).
- **Bolt origin:** still the staff tip (visible muzzle). Direction = `(aim - origin).normalize()`. Bolt visibly leaves the staff and converges on whatever the reticle is pointing at — standard 3rd-person-shooter "muzzle convergence" behaviour (Gears, Fortnite, Last of Us, Skyrim 3P all do this).
- **Backwards-compatible Y-aware extension:** `raycastXZ` callers that don't pass `originY`/`dirY` get the legacy 2D-only behaviour (footprints treated as infinitely tall). Existing callers like the projectile per-step raycast and melee cone-cast keep working without changes.
- **Files:** [src/world/awakenedCombat.ts](src/world/awakenedCombat.ts) (`reticleAimPoint` rewrite + `getTerrainHeight` field on `AttachOpts`), [src/world/collisionWorld.ts](src/world/collisionWorld.ts) (`raycastXZ` Y-aware mode), [src/ui/mountApp.ts](src/ui/mountApp.ts) (wires `h.getTerrainHeight` to the combat handle).
- **Lesson:** When you need 3D aim resolution from a 2D collision world, supplement the 2D raycast with a separate terrain heightfield walk and pick the closer hit. Y-extension via `dist3d = hit.dist / xzLen` is mathematically correct in isolation but produces aim points unbounded below ground — the heightfield walk gives you the natural 3D stop.

### Phantom-light pool — multi-instance lit objects without recompile freeze (2026-04-19)

- **Issue:** Player-placed campfires looked dead compared to the dream-mode dock-yard original. The dock-yard version casts orange light onto nearby logs / stones / ground (via two permanent phantom `PointLight`s the LPCA's `tick()` animates). The first multi-instance shipped with off-scene dummy `PointLight`s passed to `createCampfireLPCA` — flame mesh emissive still pulsed but no surface glow → "fire-shaped lava lamp".
- **Cause:** Adding fresh `PointLight`s per placed campfire would flip `numPointLights` and trigger the documented "Campfire 5-second freeze" anti-pattern (every lit material in the scene gets a synchronous recompile when the program count hashes change). Off-scene dummy lights avoid the recompile but contribute zero visible light.
- **Solution: pre-allocated pool of N phantom pairs.** `craftStationBuilder.ts` allocates 4 pairs (= 8 `PointLight`s) at attach time, parks them at `(10000, -10000, 10000)` with intensity 0. `numPointLights` is constant for the rest of the session.
  - `claimLightSlot(stationId, x, y, z)` moves a free pair to the campfire's pit position. The LPCA's `tick()` drives intensity normally — visually identical to dream-mode dock-yard.
  - `releaseLightSlot(idx)` parks the pair back off-scene + intensity 0 when the campfire is removed.
  - Pool exhaustion (5+ campfires) gracefully falls back to off-scene dummies. Those campfires still glow via emissive flame meshes + bloom, just without the surface light bath.
- **Pool size choice:** 4 covers a typical camp setup. Each slot adds 2 `PointLight`s to the per-frame lighting cost, so a larger pool would push the lighting budget. Easy to bump if needed.
- **Color / distance / decay** match the dream-mode `characterScenePreview.campfirePhantomFireLight` exactly (`0xff8833` / 2.4 / 1.35 for fire; `0xffcc66` / 1.2 / 1.8 for hot) so the visual is indistinguishable.
- **Files:** [src/world/craftStationBuilder.ts](src/world/craftStationBuilder.ts) (`STATION_LIGHT_POOL_SIZE`, `lightPool`, `claimLightSlot`, `releaseLightSlot`, integration into `createInstance` / `disposeInstance` / `dispose`).
- **Generalisable:** For any system that needs N lit instances at runtime where N is unbounded but small, pre-allocate a phantom-light pool at attach time. The phantom-light invariant (no fresh `PointLight`s after boot) stays intact; pool overflow degrades gracefully to emissive-only.

### Lock-on slow-bolt auto-hit gate (2026-04-19)

- **Issue:** Magic bolts with T-lock-on engaged would graze right past their target without registering damage. Symptom was inconsistent — sometimes the bolt hit (raycast caught the mob's footprint), sometimes it sailed through (raycast missed by 0.3 m and the bolt continued past).
- **Cause:** Bolt speed dropped to 14 m/s for the new visual feel. At 60 fps the per-frame swept XZ distance is ~0.23 m. Mob footprint radii are small (rat 0.4 m, wolf 0.55 m, wanderer 0.55 m). The per-step `raycastXZ` checks if the swept segment from `prevXZ` to `currXZ` PASSES THROUGH any footprint, which works fine for fast bolts where the swept distance dwarfs the footprint, but the slow new bolt's swept distance is comparable to the footprint radius. Real risk of grazing past without intersecting.
- **Solution: auto-hit gate for the locked target.** `magicProjectiles.update()` adds a guard inside the homing block: when the bolt is within `LOCKED_TARGET_AUTOHIT_RADIUS = 0.65 m` of the locked target's chest, damage applies UNCONDITIONALLY (no reliance on the per-step XZ raycast). 0.65 m gives generous tolerance around the target so once the bolt is genuinely on top of its locked target, the hit definitely registers.
- **Also bumped `HOMING_TURN_RATE` 3.2 → 7.5 rad/s** — the slow 14 m/s bolt was undertracking moving wolves; aggressive turn rate lets the bolt stay locked on a strafing target.
- **Why only for locked targets:** the auto-hit gate is the player's "I told the bolt to home" promise — when lock-on is engaged, the bolt MUST hit. Free-aim shots still rely on the per-step raycast (the player accepted the convergence trade-off when they didn't lock on).
- **Files:** [src/world/magicProjectiles.ts](src/world/magicProjectiles.ts) — `LOCKED_TARGET_AUTOHIT_RADIUS` + `HOMING_TURN_RATE` constants, auto-hit branch inside the homing block.
- **Lesson:** Slow projectiles + small targets need an auto-hit fallback when the per-frame swept distance is comparable to the target's hit radius. The per-step raycast pattern works fine for fast hitscan-like projectiles but undertracks for "magic missile" speeds.

### GoE-style flush snap — same-kind lateral always wins (2026-04-18)

- **Issue:** Placing a floor next to an existing floor stacked the new floor on top (bottom-of-new at top-of-existing, ~5 cm step up) instead of extending flush at the same Y level. Same symptom for ceilings, foundations, and other "ground tile" pieces.
- **Cause:** The choose-between rule between lateral snap (`findSnapXZ`) and top-stack snap (`findSnapY`) used a "closer-to-cursor wins" heuristic with very loose thresholds — 3.0 m euclidean for lateral and 4.5 m manhattan for top. When you placed a floor near an existing one, the foundation underneath the existing floor exposed a top snap that accepts `floor` (manhattan distance ~1.5 m → matched), and at certain cursor angles its `topDist` could beat `lateralDist`. The new floor stacked on the foundation top instead of extending flush.
- **Solution (per `gameofempiresDocs/docs/BUILDING_SYSTEM.md` — the GoE spec we're aligning to):**
  - **Same-kind lateral always wins.** When `findSnapXZ` matches a placed piece of the same kind (floor next to floor, wall next to wall, ceiling next to ceiling, etc.), the top-snap competition is skipped entirely. This is GoE's "deck extension" rule. Implemented via a new `sameKind: boolean` field on the `findSnapXZ` return shape; the `update()` choose-between checks it first.
  - **Tighter lateral threshold.** Replaced `Math.max(3.0, halfExtent)` with `LATERAL_SNAP_DIST = GRID_SIZE * 1.5 = 2.25 m` — exactly what the GoE doc specifies (*"within 1.5× grid distance of an existing piece's edge"*). No more snap-from-across-the-room.
  - **Tighter, euclidean top-stack threshold.** Replaced `SNAP_THRESHOLD = 4.5` (manhattan) with `TOP_SNAP_DIST = 1.0` (euclidean). Top-stack only competes when the cursor is genuinely OVER the placed piece's footprint — matches GoE's *"ghost OVERLAPS top snap point"* intent.
- **Why this priority order is correct:** Top-stack is for OBVIOUS vertical attachments — wall on foundation top, ceiling on wall top, roof on ceiling. The cursor is genuinely on top of the existing piece. Lateral extension is for LIKE-pieces extending sideways — floor row, wall row, ceiling tile grid. The cursor is next to the existing piece. When both could match, lateral SHOULD win for same-kind pairs because that's "extend the deck" — the GoE design intent. Closer-to-cursor was a generic heuristic that happened to fail this case.
- **Files:** [src/world/buildModeController.ts](src/world/buildModeController.ts) — three edits: constants block (`LATERAL_SNAP_DIST`, `TOP_SNAP_DIST`, deprecated `SNAP_THRESHOLD`), `findSnapXZ` (added `sameKind` to return shape), `findSnapY` (manhattan → euclidean + tighter threshold), `update()` choose-between logic.
- **Reference:** GoE building system spec lives at `gameofempiresDocs/docs/BUILDING_SYSTEM.md` (external repo). When extending the snap pipeline, treat that doc as the authoritative behavior spec — the implementation in `buildModeController.ts` is the port.

### Tree-top landing height — measure mesh, don't trust constants (2026-04-18)

- **Issue:** Player feet landed too high above the visual canopy on medium / large trees. Smaller trees were perfect; the error scaled with the per-instance `sm` so a sm≈2 medium tree had the player standing about 1 body length above the actual leaves. Same issue for the awakened-mode apple-tree scatter.
- **Cause:** Both harvest paths registered the collision footprint's `topY` from a hand-tuned per-species/per-shape constant (`TREE_SPECIES_TOP_Y_PER_SM` in `forestEnvironment.ts`, `HARVEST_TOP_Y_OFFSET` in `freeRoamHarvestNodes.ts`) multiplied by the per-instance scale. Those constants were rough estimates that didn't match the actual LPCA-built mesh height; the mismatch scaled linearly with `sm` so it became dramatic on big trees.
- **Solution — measure the mesh, don't trust the constant.** Same idiom Unity / Unreal use to derive collision capsule heights from rendered meshes:
  - **`dockForestBatchedScene.ts` `VariantTemplate.maxYAtUnitScale`:** after `mergeByMaterial` + `computeBoundingBox()` per sub-mesh (already done for frustum culling), take `max(subMesh.boundingBox.max.y)` across the variant. Each instance's collision `topY` becomes `spec.baseY + variant.maxYAtUnitScale * spec.scale` instead of the prior `spec.topY` (which used the hand-tuned constant). Per-variant measurement (3 variants per tree species) means each variant's actual canopy max drives its instances' landing surface.
  - **`freeRoamHarvestNodes.ts` `KindHandle.maxYAtUnitScale`:** measure the same way after the per-kind merged template is built. Instance scatter + respawn re-register + grow finalize all use `h.maxYAtUnitScale * scaleBase` for `topY`. Magic-crystal external-Group keeps its hand-tuned 1.6 m (no merged mesh available; was already correct per user feedback).
  - 1.0 m fallback if the bounding box is degenerate (`!Number.isFinite`) — defensive, keeps collision sane while making the bug obvious.
- **Why per-variant (not per-species):** dock forest builds 3 LPCA-randomized variants per tree species. Each variant has slightly different geometry (different branch counts, leaf depth) so the canopy max differs. Sharing one species-level constant would re-introduce the same drift on a smaller scale. Per-variant is essentially free since `boundingBox` is already computed for `addGeometry()`.
- **Files:** [src/world/dockForestBatchedScene.ts](src/world/dockForestBatchedScene.ts) (`VariantTemplate.maxYAtUnitScale` field, computed in `buildVariantFromTemplate`, used as `measuredTopY` in scatter loop), [src/world/freeRoamHarvestNodes.ts](src/world/freeRoamHarvestNodes.ts) (`KindHandle.maxYAtUnitScale`, used in scatter + respawn re-register + grow finalize).
- **Lesson for future LPCA collision work:** any time a hand-tuned size/height constant is duplicating data the LPCA mesh already knows about, just measure the mesh. Drift is inevitable with constants because the mesh evolves but the constant doesn't.

### Auto step-up + predicted-XZ landing — walk on floors / land on trees (2026-04-18)

- **Issue 1 (walk on floor):** Cabin floor pieces (`topY = y + 0.05`) and foundations (`y + 0.15`) blocked the player like walls. Player feet at terrain Y=0; floor topY=0.05 was inside the player's Y-band → `inYBand` returned true → push-out applied. Even if the player got past collision, `syncAvatarFeetToTerrain` re-snapped `avatar.y = terrainY` every frame, undoing any step-up.
- **Issue 2 (land on tree):** The airborne vertical substep always sampled `getGroundY` at `avatar.position.x/z`, which holds the PRE-horizontal XZ at frame start. When jumping forward toward a tree, the substep never sees the tree (player isn't over it yet). Then horizontal phase moves them past — by the next frame they've already fallen below tree top and the rejection clause `top > currentY + GROUND_TOLERANCE` skips the tree. They fall straight past it.
- **Cause:** Two separate Y-aware-collision blind spots that sit at the same architectural seam (vertical-then-horizontal phasing + terrain-only foot-snap). Standard 3D action games solve both via two well-known patterns: **auto step-up** (Unity CharacterController `stepOffset`, Unreal `MaxStepHeight`, Source engine 18-unit step) and **continuous collision detection during fall** (sweep test or predicted-position landing check).
- **Solution (three coordinated changes):**
  1. **Step-up in `collisionWorld.inYBand`:** new `stepUp` parameter (default 0). When `obstacle.topY <= playerBottomY + stepUp + epsilon`, the obstacle is skipped by horizontal push-out. `freeRoamControls` passes `STEP_UP_HEIGHT = 0.55` while grounded (matches stair top exactly so a single press-W climbs) and 0 while airborne (no auto-climb mid-fall — that's what the predicted-XZ lander handles).
  2. **`getGroundY` accepts `snapUpHeight`:** old behavior used a single 10 cm tolerance which rejected anything more than that above feet. Caller now passes `STEP_UP_HEIGHT` for grounded snap-up so foundations / stairs / low rocks count as ground; airborne lander still uses the 10 cm landing slop so the player doesn't get yanked up onto trees mid-fall.
  3. **Surface-Y provider on the dock's foot-snap:** `characterScenePreview.syncAvatarFeetToTerrain` now consults a wired `surfaceYProvider` (set to `() => freeRoamHandle.getGroundedSurfaceY()` from `mountApp`). When non-null, that surface Y replaces terrain Y in the foot snap. `freeRoamControls` writes `lastGroundedSurfaceY` from the post-move `getGroundY` sample, so the dock and the controls are always in agreement about "where are the feet?". Cleared (back to terrain-only) on realm flip back to deck.
  4. **Predicted-XZ in airborne lander:** `sampleGroundY` now also samples at `(avatar.x + velX*dt, avatar.z + velZ*dt)` and takes the higher of the two. When jumping forward toward a tree, the substep sees the tree on the predicted XZ even though the current XZ isn't over it yet. This is the cheap version of full continuous collision detection (Source/Unity sweep tests); fine for our scale.
- **Why predicted-XZ is the right pattern (vs. swapping phase order):** Both vertical-first AND horizontal-first orderings have failure modes. Vertical-first (current): can't catch trees the player is about to reach. Horizontal-first (previous): when player feet drop below tree.topY mid-fall, the Y-band push slides them sideways OFF the tree before vertical lander snaps them on top. Predicted-XZ keeps vertical-first (which solves the slide-off bug) AND solves the look-ahead bug — strictly better than either ordering alone.
- **Tuning:** `STEP_UP_HEIGHT = 0.55` matches stair top exactly. Industry references: Unity CharacterController default 0.3, Unreal CMC default 0.45, Minecraft 0.6, Source engine 0.46 — our 0.55 sits at the high end of the sane range and explicitly accommodates the cabin's 0.55 m stair piece. Walls (2.4 m), doors (1.8 m), trees (1.6 m+) all far exceed step-up so they still block normally.
- **Files:** [src/world/collisionWorld.ts](src/world/collisionWorld.ts) (`inYBand` step-up gate, `getGroundY` snap-up param, `DEFAULT_STEP_UP_HEIGHT` export), [src/world/freeRoamControls.ts](src/world/freeRoamControls.ts) (`STEP_UP_HEIGHT`, `lastGroundedSurfaceY`, `getGroundedSurfaceY()` API, predicted-XZ in `sampleGroundY`, snap-up after walk-off check), [src/visual/characterScenePreview.ts](src/visual/characterScenePreview.ts) (`surfaceYProvider` + `setSurfaceYProvider()` + collision-aware `syncAvatarFeetToTerrain`), [src/ui/mountApp.ts](src/ui/mountApp.ts) (wires + clears the provider on realm flip).
- **Open known-limit:** Giant trees (sm ~ 4-5, topY 6-8 m) exceed the player's max apex (~6 m with double-jump). Smaller/medium trees (apple, birch, pine, balsam at sm ≤ 2) are now landable. If "land on giants" becomes a goal, the answer is taller jump physics OR climbable trunks — not a collision tweak.

### Universal sapling-grow respawn cycle (2026-04-18)

- **Goal:** Every harvestable resource respawns AND visibly grows from a sapling/pebble/sprout up to its full mature size — not the prior 7-min "snap back at full scale" cycle.
- **Architecture:** Symmetric counterpart to the existing break archetypes. New `GrowAnim = 'tree_grow' | 'bush_grow' | 'stone_form' | 'fiber_grow' | 'crystal_emerge'` (the last one only in the awakened-mode scatter; dock forest has no crystals). Each grow archetype lerps the per-instance scale from a per-archetype minimum (e.g. `TREE_GROW_MIN_SCALE = 0.08`, `STONE_FORM_MIN_SCALE = 0.30`) up to 1.0 via `easeOutCubic`, multiplied by the node's per-spec `scale` so a giant oak grows from a tiny seedling all the way to its giant size. Stone gets a small settling wobble in the first half of the animation; the rest are pure scale-up. Crystals run on the external Group (same path as the existing `crystal_flash`).
- **State machine per node:**
  1. Mature: `hp = maxHp, isGrowing = false, availableAtMs = 0`. Visible at full scale, harvestable, collision registered.
  2. Broken: `hp = 0, isGrowing = false, availableAtMs = now + 180 s`. Invisible (`setVisibleAt(false)` or `HIDDEN_MATRIX`), no collision. The respawn wait was cut from 7 min to 3 min — half-time gain is necessary because the grow phase adds 12-60 s of "visible but invulnerable" time on top.
  3. Wait elapsed -> respawn loop pushes a grow anim, sets `isGrowing = true, availableAtMs = 0`. Instance becomes visible immediately at `node.scale * minScale` (sapling) so the first frame reads as a believable seedling.
  4. Grow done: animation finalizer snaps to base matrix at full scale, restores `hp = maxHp`, registers collision at full radius, clears `isGrowing`, removes from `brokenNodeIndices`. Node is back to mature state.
- **Why no collision during growth:** A 0.08-scale sapling has a 1.5 cm trunk radius — registering the full footprint there would block the player from a spot that visually appears empty. Saplings are non-blocking by design. Player can also walk through ferns/grass mid-grow same as mature ones.
- **Why no harvest during growth:** A sapling can't yield wood. Both `getNodeNearAvatar` and `applyHit` gate on `isGrowing` (or rely on `hp == 0` which stays true throughout growth). Player has to wait for full maturity.
- **Total cycle time per kind:** trees 3 min wait + 60 s grow = ~4 min. Bushes/berry/heather 3 min + 30 s. Rocks/ore 3 min + 12 s. Fern/grass/vine/moss 3 min + 18 s. Crystal 3 min + 14 s.
- **Files:** [src/world/dockForestBatchedScene.ts](src/world/dockForestBatchedScene.ts) (BatchedMesh path; `setMatrixAt` per frame), [src/world/freeRoamHarvestNodes.ts](src/world/freeRoamHarvestNodes.ts) (InstancedMesh + external-Group path).
- **Performance note:** The 7 prior optimizations are all preserved — `brokenNodeIndices` set still drives the per-frame respawn scan (typically 0-5 nodes, not 360), reusable scratch matrices, no per-frame allocations in the grow path. Active animations grew from "0-3" to "0-15" peak (one per concurrently-growing node) but the matrix math is the same compose/setMatrixAt as before; no measurable cost.

### Dock-forest BatchedMesh refactor — wind + fall + variation + low draw (2026-04-18)

- **Goal:** Every visible plant on the dock should sway in the wind AND be harvestable AND have a fall/collapse animation on harvest. Trees keep per-tree LPCA variation (multiple shapes per species). Draw calls stay low.
- **Architecture:** `THREE.BatchedMesh` (Three.js r182) replaces the prior InstancedMesh-per-species path. BatchedMesh delivers all five constraints simultaneously: (1) low draw calls — one per material across all kinds + variants; (2) per-instance hide via `setVisibleAt(i, false)` (first-class API, not a zero-scale hack); (3) per-instance fall animation via `setMatrixAt(i, animatedMatrix)`; (4) per-tree LPCA variation via multiple `addGeometry()` calls per (kind, species) — instances pick a variant; (5) wind sway via the path-aware shader extension below.
- **Path-aware wind shader (`USE_BATCHING` / `USE_INSTANCING` / merged):** [src/visual/idleCraftVegetationWind.ts](src/visual/idleCraftVegetationWind.ts) `installVegetationWindOnMaterial` derives the world-space wind phase using whichever transform pipeline applies. For BatchedMesh: replicate `getBatchingMatrix(getIndirectIndex(gl_DrawID))` ourselves (functions live in `<batching_pars_vertex>` which IS in the parsing block, available globally — even though `<batching_vertex>` runs AFTER our `<begin_vertex>` injection). For InstancedMesh: read `instanceMatrix` directly. For merged: `transformed` IS world space because `mergeByMaterial` bakes the transforms. Result: every plant sways with per-instance world-space phase (looks organic, not in-unison). Cache key bumped to `vegWind4` to invalidate older programs.
- **Per-vertex bake helper for templates:** `bakeVegetationWindHeightForTemplate` (new) bakes `vegetationWindH` on a unit-scale template Group via local-space Y. The InstancedMesh / BatchedMesh skip in `bakeVegetationWindHeightAboveGround` is preserved (their geometries are template-shared); the new helper is the explicit way to bake the attribute on a template before wrapping it in `addGeometry`.
- **New module: [src/world/dockForestBatchedScene.ts](src/world/dockForestBatchedScene.ts)** (~700 lines):
  1. Group specs by `(kind, species/variant)` bucket.
  2. For each bucket: build N LPCA-randomized templates (3 for trees, 2 for shrubs/berries/ferns/heather, 1 for rocks, 1 unique geometry per patch).
  3. Run `mergeByMaterial` on each template -> sub-meshes per material.

### Universal harvest + tree-top landing fix (2026-04-18)

- **Issue 1 (landing):** After the universal-collision pass, the player still couldn't stick a landing on top of birch / apple / oak / pine canopies. ### First-sunset hard freeze — directional shadow light count churn (2026-04-16)

- **Issue:** Hard 2–10 second freeze **only on the first in-game sunset**, ~3–5 seconds **after** the sun plasma disc faded out and **well before** the moon rose above the horizon. Everything stopped (RAF + UI). Second-night sunset onward was fine. (Supersedes the earlier "Dusk/dark transition hitch — shadow gap" entry from 2026-04-14: the 0.5-complementary-threshold split closed the handoff gap but an **additional** gap — `keyLight.castShadow` dropping via `dayMix > 0.12 && sunDirectFrac > 0.07` guards before `moonLight.castShadow` could rise — still existed.)
- **Cause:** Three.js `WebGLPrograms` hashes lit-material shader programs by `numDirLightShadows`. At first sunset, the count walked **1 → 0 → 1** because both directional lights (`keyLight` / `moonLight`) briefly had `castShadow = false` at the same time. The `0 → 1` transition forced Three.js to **recompile every lit `MeshStandardMaterial` in the scene** (terrain, trees, rocks, avatar, fairies, fungi, water, equipment, …) synchronously on the main thread. First-time-only because shader programs stay in `WebGLPrograms` cache for the session.
- **Why earlier warm-up passes didn't help:** `renderer.compile()` on boot only produces programs **for the current lighting state**. The warm pipeline ran with `keyLight.castShadow=true, moonLight.castShadow=false` (1 caster, sun hash) and never produced the `0 caster` or `moon-caster` variants. Later attempts to precompile a "night variant" by flipping `moonLight.castShadow = true` during warm-up compiled the **wrong hash** (two casters) and still didn't cover the transient 0-caster gap.
- **Solution:** Both directional lights keep **`castShadow = true` for the entire session**. `numDirLightShadows` is constant → one program hash → no recompile, ever. Below-horizon contributions are invisible because Three.js multiplies the shadow factor by the light contribution, and both lights' `intensity` fades to 0 below the horizon (via `sunKeyIntensity` / `moonStrength`). Cost: one extra 1024² depth pass per frame for whichever light is currently "off-duty" (~0.5–2 ms on modern GPUs).
- **Removed with this change:** `shadowMoonHandoffSmooth`, `useMoonShadow` / `useSunShadow` logic, the `dayMix > 0.12 && sunDirectFrac > 0.07` guards, and the two-pass day/night variant warm-up (`warmCelestialPrograms` is kept as a no-op stub for compatibility — safe to delete later).
- **Related "keep always visible" changes (moon disc, fairy dust trail):** In this pass, `moonDisc.visible = true` and the `FairyDustTrail.setVisible` toggles were also removed — `uOpacity` / trail `fade=0` (additive × 0 = invisible) gate the visuals in-shader. **Watch for** the previously-noted horizontal-line artifacts on the night sky from near-zero-opacity draws (see 2026-04-14 "Dusk/dark transition hitch — shadow gap"); if they reappear, revert those two `visible = true` changes — they are **not** what fixes the freeze (the shadow fix above is).
- **Do NOT try:** (1) toggling either directional light's `castShadow` based on altitude or `dayMix` thresholds — it **will** reintroduce the `1 → 0 → 1` program hash churn; (2) adding a third "0-caster" warm pass — Three.js still has to produce lit-material programs for the transient state, and any light-count flip at runtime will still block the frame.
- **Diagnosing this pattern again:** Hard freeze at an in-game time that repeats, first-occurrence-only, with console quiet and GPU pegged → suspect a **WebGL program hash change** (light counts, define flips, envMap presence, material swaps). Use `idleCraft.perfDockFrame` (see "Dock frame pipeline + perf probes") to rule out JS hot paths first; if slices are all short but RAF still stalled, it's a GPU-side compile / link.
- **Files:** `src/world/idleCraftDockEnvironment.ts`, `src/visual/characterScenePreview.ts`, `src/visual/idleCraftNightMagicLPCA.ts`

- **Issue (freeze):** First time the player pulled out a held torch in awakened mode (typically when night fell or when crafting `r_torch`), the dock froze for ~5 seconds — exact same symptom and feel as the original 2026-04-17 campfire 5-second freeze.

### Dock preload split: chunks during cutscene, heavy work on title mount (2026-04-17)

- **Original change (rolled back same day):** First version of this fix called the full `schedulePreloadAfterPaint` (which triggers `startIdleCraftDockPreload` → `CharacterScenePreview.create` → constructor + GPU warm) right after splash dismiss, on the assumption that the browser decodes `<video>` on a separate thread and the main-thread constructor work wouldn't stutter the cutscene. **It did.** The user reported a hard freeze around the "Mira, my Mira" narration beat. ### Dock preload: staged-phase constructor + rIC warm to spread the work across the boot flow (2026-04-17)

- **Issue:** Even after the chunk-only-prefetch fix from earlier the same day, the `CharacterScenePreview` constructor was still a single ~300-500ms synchronous block. ### Awakened-mode FPS — cross-instance scatter merge + per-frame allocation cleanup (2026-04-18)

- **Issue:** Awaken mode hovered around 50-70 fps on mid-tier hardware. The free-roam harvest scatter (`src/world/freeRoamHarvestNodes.ts`) was emitting ~150-220 draw calls on its own — well over the plan §9 budget of "~50-150 instances should add ≤5 draw calls."
- **Cause:** The original module merged each instance individually: every apple tree / fiber tuft / berry bush / rock pile / ore chunk got its own `mergeByMaterial` pass producing 1-4 meshes (one per material), then those wrapped meshes were added to the scene as separate children. Across 70+ scattered nodes that's 70 × per-instance-mat-count = **150-220 separate scene meshes / draw calls**, even though all instances of a kind share materials (the LPCA tree/bush builders pull from `physLeafCache` / `stdLitCache` / `physGlossCache`; simple shapes use a single per-kind `sharedMat`). The instance-level merge was working against itself — `mergeByMaterial` only collapses across the meshes it sees in one Group, so per-instance calls couldn't collapse across instances. Per-instance `.visible` toggles for harvest-cooldown hiding were the design constraint that locked us into the per-instance merge pattern. Additionally, `mountApp.ts`'s render loop allocated `new Set(state.unlockedCardIds)` and a fresh `{x, z}` proximity object every frame, plus called `scenePreview.getFreeRoamHandles()` (returns a fresh object literal) on the proximity scan AND in the E-handler.
- **Solution:**
  - **Cross-instance per-kind merge.** `freeRoamHarvestNodes.ts` now builds **every instance of a kind** into one staging Group, runs `mergeByMaterial` ONCE across the whole staging group, and adds the merged result to a per-kind sub-Group. Because the LPCA builders share material instances and the simple shapes share `sharedMat`, the merge collapses to 1 mesh per unique material across all instances — wood ships in ~3-4 draws (apple-tree's bark + leaves + accents) regardless of how many trees, stone in 1 draw across all 14 piles, each ore tier in 1 draw, etc. **Total scatter draw cost: ~25-35 calls (down from 150-220, a ~6× reduction).**
  - **UX trade-off (locked):** harvested nodes no longer visually disappear during the 45 s cooldown — they stay in the merged kind mesh and the cooldown is purely logical (`availableAtMs` gates `getNodeNearAvatar`, so the "Press E" prompt vanishes and a second E press won't fire). Same UX as the dock's deck-mode harvest source meshes (which never visually delete either). The inventory grant + bag flash are the player's confirmation. This UX is what unlocks the cross-instance merge — individual `.visible` toggles would require breaking the merge.
  - **Per-kind visibility.** Deck-mode unlock gating is now a single `kg.visible = bool` flip per kind sub-Group instead of N flips across N node meshes — 13 ops per rule application instead of 70+.
  - **Signature-gated rule application.** `setVisibilityRules()` builds a `realm|forceShowAll|sortedCardList` signature and early-returns when unchanged. `mountApp.ts` `subscribe`s the store and calls this only on emit (cheap join + compare); the per-frame loop carries zero rule-application cost.
  - **Per-frame allocation cleanup in `mountApp.ts`:** (1) `cachedFreeRoamHandles` caches the `getFreeRoamHandles()` object once per session, killing 2 fresh object literals per awakened frame; (2) `proximityScratch = {x, z}` is reused across the scan + E-handler instead of `{ x: ax, z: az }` literals; (3) the visibility `Set` is pre-allocated and only refilled when `unlockedCardIds.join('|')` changes; (4) prompt `textContent` writes are gated on actual text change so a steady-state "Press E to harvest wood" doesn't reflow the DOM at 144Hz; (5) deck mode skips the proximity scan entirely.
  - **Shadow trim.** Per-kind shadow policy: trees + berry bushes cast (their silhouettes are read at dock scale and shadow grounds them); fiber tufts, stone piles, ore chunks do NOT cast (small ground-bound props where the depth-pass cost outweighs the visual gain). `receiveShadow = false` on all merged scatter (terrain handles ground shadow reception). Cuts ~40-50 entries from the directional shadow pass.
- **Result:** Free-roam scatter draws collapse to ~25-35 (was ~150-220). On a Ryzen 5700G + iGPU laptop test the awaken-mode steady-state goes from ~58-72 fps to ~110-130 fps; on a desktop RTX 3060 it pegs at the 144Hz cap. No visual regression beyond the "harvested node stays visible" UX trade-off (which actually reads MORE consistent with deck mode).
- **Files:** `src/world/freeRoamHarvestNodes.ts` (full rewrite — cross-instance merge + signature-gated visibility + harvest-persists UX), `src/ui/mountApp.ts` (cached handles, scratch proximity object, store-subscribed rule application, prompt write gating, deck-mode early-out).
- **Do NOT do:** (1) Re-introduce per-instance `mergeByMaterial` calls in the scatter loop — the win is exactly the cross-instance merge. (2) Add per-instance `.visible` toggles to the harvest scatter — it'd require rebuilding the merge on every toggle (catastrophically slow) OR breaking the merge entirely (back to 150-220 draws). If we ever need real harvest-time visual feedback, layer it as a separate temporary effect (particle puff, faded depleted-overlay decal, etc.) instead of toggling the static merged geometry. (3) Call `scenePreview.getFreeRoamHandles()` in the per-frame render loop — it builds a fresh object literal each call. Use the cached handle.

### Awaken-mode FPS round 2 — spatial-tile vegetation + daylight night-magic skip (2026-04-18)

- **Issue:** After the per-instance-merge cleanup (entry above), awaken mode still capped around 70-85 fps on mid-tier hardware — well short of the 100+ target. Profiling pointed at GPU vertex/fragment shading on the merged forest, not draw-call count.
- **Cause — bounding sphere covers the whole map.** `mergeByMaterial(treeWorldNear)` collapses every tree in the disk into a few merged meshes whose bounding spheres cover the **entire** map disk. Three.js's auto frustum culling never kicks in even though only ~25-40% of trees are in the camera's view at any moment, so the GPU vertex-shades + rasterizes every tree on both sides of the avatar each frame. The vegetation wind shader runs per vertex over the full forest. Same problem on the harvest scatter (which had been fixed for draw count in the prior entry but still had a single map-spanning bounding sphere per kind).
- **Cause — night magic field churn during the day.** `IdleCraftNightMagicField.update()` iterates every emissive material (scaling `emissiveIntensity` to ~0), every patch point light (intensity to ~0), and every drifting mote (recomputing orbital positions even though everything is invisible) every single frame — even when the player is in awakened mode at noon. Pure CPU waste.
- **Solution — spatial-tile merge.** New helper `src/visual/spatialTileMerge.ts` exports `mergeByMaterialTiled(staging, gridSize, originX, originZ, cellSize)`: buckets staging children into an N×N world-XZ tile grid before running `mergeByMaterial` per tile. Each tile's merge has a **tight per-tile bounding sphere** (~1/N² of map area) so frustum culling drops 60-75% of vegetation tiles per frame. The shadow-camera frustum (which covers ~half the disk around the avatar target) culls ~half the tiles in the depth pass too. Material identity is preserved across tiles (`mergeByMaterial` buckets by material reference, and we tile BEFORE merging so shared cached materials end up in multiple tile merges) → the vegetation wind shader's WeakSet-gated material patch + shared `uVegetationWindTime` uniform still apply once / update once regardless of tile count. Wired into `forestEnvironment.ts` (3×3 grid for `mergedForestNear` / `mergedForestFar` / `mergedUnderstory`) and `freeRoamHarvestNodes.ts` (3×3 grid per kind).
- **Solution — daylight night-magic skip.** `IdleCraftNightMagicField.update()` now early-returns when both the smoothed display strength AND the target visibility are < 0.003 (full daylight). On the first such frame it latches a one-shot "everything to dormant" pass that zeros all emissive intensities and patch-light intensities, then bails. Subsequent daylight frames hit the latched fast-path and skip the loops entirely. The fairy swarm's own zero-ratio bail (`activeCount === 0 && ratioSmoothed < 0.008`) handles its smoothed ratio decay so we still call into it once per frame (cheap ratio-only update). Latch resets when visibility ramps back up at dusk so the next sunset re-flushes cleanly.
- **Why this respects the LEARNINGS gotchas:** (1) tile merging does NOT toggle `.visible` on stars / moon disc / plasma sun (those were the `numPointLights` / `numDirLightShadows` recompile freezes — frustum culling is a different path that doesn't change shader programs); (2) tile material clones from `cloneMaterialsForVegetationWind` are still WeakSet-deduped by `installVegetationWindOnMaterial`, so the wind shader is patched once per material across all tiles; (3) the daylight skip never touches `.visible` on the night-magic root or fairy swarm — both stay scene-attached so first-use compile cost was paid at boot.
- **Result:** On a Ryzen 5700G + iGPU laptop (the test machine that was at 110-130 fps after the previous round), awaken mode now hits the 144 fps cap at noon with headroom and stays > 110 fps at twilight (when the night magic ramp kicks in). Vertex shader work for vegetation wind drops by ~60-75% (tiling), shadow pass cost drops by ~50% (tiling + per-tile shadow-frustum culling), and per-frame CPU cost during day drops by the entire night-magic field update. **No visual regression** — frustum culling is conservative (a tile is drawn the moment any pixel of its bounding sphere touches the frustum), so vegetation never pops at the screen edge, and the day-skip only fires when everything would have been at zero intensity anyway.
- **Files:** `src/visual/spatialTileMerge.ts` (new helper), `src/visual/forestEnvironment.ts` (3×3 tile-merge for the 3 forest groups), `src/world/freeRoamHarvestNodes.ts` (3×3 tile-merge per kind), `src/visual/idleCraftNightMagicLPCA.ts` (`dayQuietApplied` latch + early-return in `IdleCraftNightMagicField.update`).
- **Do NOT do:** (1) Tile too aggressively (>4×4 grids) without measuring — beyond ~9 tiles per group, draw-call growth starts to outpace culling savings on systems that aren't fragment-bound. 3×3 is the sweet spot for the dock map size. (2) Set `frustumCulled = false` on a tile merge to "fix" a culling pop — it'd defeat the entire purpose. If wind-shader vertex displacement ever gets aggressive enough to push vertices outside the baked bounding sphere, expand the sphere (`mesh.geometry.boundingSphere.radius += MAX_WIND_AMP`) instead. (3) Toggle the night-magic root's `.visible` based on the day-skip latch — same first-use compile freeze pattern documented in the campfire / sunset entries above. The early-return only skips JS work; the renderer still sees the dormant-but-attached subtree.

### Awaken-mode FPS round 3 — fullscreen DPR cap + post-stack trim (2026-04-18)

- **Issue:** After rounds 1 + 2 (per-instance merge cleanup, spatial-tile vegetation, daylight night-magic skip), embedded preview hit ~80-90 fps but **fullscreen awakened mode dropped to 30-40 fps**. The user nailed the diagnosis: the prior optimizations addressed vertex-shader / draw-call / CPU work, but going from a small embedded canvas (~200-400 K pixels per frame) to fullscreen (1080p × DPR 2 ≈ 8 M pixels per frame) is a **20-40× pixel-fillrate jump**. Fragment shading + post-processing scale linearly with pixel count, so when the GPU is fragment-bound NOTHING vertex/CPU-side moves the needle.
- **Cause:** Two compounding costs at fullscreen + HiDPI: (1) the renderer's pixel ratio multiplied with the canvas CSS size produces an enormous render buffer (high tier caps DPR at 2.25, so a HiDPI laptop renders at 4-8 MP per frame); (2) the post-processing stack runs SSAO (kernel size 12 × resolution scale 0.85 — multiple texture taps per pixel) AND bloom (5+ downscale + blur passes) on top of every frame, each adding several full-screen passes. Combined, fragment shading dominates frame time and no amount of vertex-shader culling helps.
- **Solution — per-realm render budget.** New `setAwakenedRenderBudget(awakened: boolean)` method on `CharacterScenePreview`, wired from `setAwakenedFreeRoam` so it auto-applies on realm flip:
  - **DPR hard-cap at 1.0** when awakened. A new `computeEffectivePixelRatio()` helper centralizes the pixel-ratio math used by both the constructor and `onResize`. On a 1080p × DPR 2 panel the render buffer drops from 7.95 MP to 2.07 MP — **~4× less fragment-shader work** with minimal visual regression because (a) the camera is moving in awakened mode (motion masks aliasing), (b) FXAA stays in the post stack to soften any introduced jaggies, and (c) most consumer panels don't actually display sub-CSS-pixel detail.
  - **Drop bloom + SSAO from the post stack** when awakened. `applyDockPostProcessing()` now overlays `{ bloom: false, ssao: false }` onto the resolved options when the awakened budget is on. FXAA + vignette stay (each is one cheap full-screen pass at the now-reduced pixel count). Bloom strength was 0.04 (barely perceptible glow) and SSAO at kernel 12 / resolution 0.85 was the single most expensive fragment pass in the pipeline. Visual regression is small; perf gain is large.
  - **Power-user opt-out:** `localStorage.setItem('idleCraft.awakenedFullQuality', '1')` (read once at boot, cached) bypasses the budget so beefy desktop GPUs that don't need the savings keep full quality at fullscreen.
- **Why this respects the existing LEARNINGS gotchas:** (1) `setPassEnabled('bloom', false)` / `setPassEnabled('ssao', false)` toggles the pass enabled flag inside the persistent `PostProcessingStack` instance — it does NOT dispose the underlying GL programs, so toggling back on at the next realm flip doesn't trigger a first-use compile freeze; (2) DPR changes via `renderer.setPixelRatio()` resize the buffer but don't change shader programs, so no `numDirLightShadows` / `numPointLights`-style hash flip; (3) the budget is idempotent and gated on state change, so spurious calls during boot or rapid realm flips don't churn.
- **Result:** On the user's mid-tier laptop (was 30-40 fps fullscreen): expect **3-5× FPS improvement at fullscreen on integrated/laptop GPUs** (DPR 4× win + post-stack 1.5-2× win, multiplicative because fragment work is the bottleneck). Going from 30-40 fps to 100-150+ fps in fullscreen awakened mode. Embedded preview perf is unchanged (budget only kicks in for awakened realm).
- **Files:** `src/visual/characterScenePreview.ts` (`awakenedRenderBudget` field, `awakenedFullQualityOptOut` cached at boot, `computeEffectivePixelRatio()` helper, `setAwakenedRenderBudget()` method, `onResize` + constructor pixel-ratio sites refactored to use the helper, `applyDockPostProcessing` awakened-aware pass override, `setAwakenedFreeRoam` calls `setAwakenedRenderBudget`).
- **Do NOT do:** (1) Try to "fix" awakened-mode FPS by optimizing vertex / CPU paths first when the diagnostic is clearly fragment-bound (small box = fast, fullscreen = slow). The pixel-fillrate jump dwarfs every other optimization. Always check pixel count × per-pixel cost before optimizing geometry. (2) Drop FXAA along with bloom + SSAO — at the reduced 1.0 DPR cap, FXAA is what hides the aliasing introduced by NOT oversampling. Removing it makes the perf gain visible as crawling jaggies. (3) Cap DPR below 1.0 (e.g. 0.75) — sub-CSS-pixel rendering forces the browser to upscale, which looks soft and tinted. 1.0 (one render buffer pixel per CSS pixel) is the sweet spot.

### Awakened-mode periodic freeze + canvas blur — `document.startViewTransition` capturing the 3D viewport (2026-04-18)

- **Issue:** After rounds 1-3 fixed the steady-state fps, awakened mode developed a new symptom: every 1-3 seconds the scene would visibly **freeze for ~250 ms, blur, and look like a "scene reset"** (similar visual to the gather-clip end where the avatar pose-resets). The freeze was independent of FPS — it happened at 144 fps just as much as 30 fps.
- **Cause:** `mountApp.ts`'s `renderPage()` wrapped page rebuilds in `document.startViewTransition(callback)` for smooth nav animations in deck mode. The View Transitions API works by:
  1. Snapshotting the entire viewport (the dock canvas included) into an offscreen texture
  2. Running the callback to mutate the DOM
  3. Snapshotting the new viewport
  4. **Cross-fading old → new for ~250 ms with the canvas frozen behind both snapshot layers**

  In deck mode this was fine — the canvas is a small embedded preview and the cross-fade animates the surrounding deck UI. **In awakened mode the canvas is fullscreen**, so the cross-fade animation literally captures the live 3D scene, freezes it, and replays it through a fade — looking exactly like a "scene reset / freeze / blur."

  Compounding it: the store-subscriber called `renderPage()` on **every store emit while `page === 'awakening'`** (which is what the page is set to when the player breaks the spell into awakened mode). Active-gather ticks, day/night updates, hunger/thirst decay, weather changes — every single one triggered another View Transition snapshot+freeze. At ~1-3 emits/second during normal play, the player saw a freeze every ~1-3 seconds.
- **Solution:** Single surgical fix in `src/ui/mountApp.ts`:
  - **`renderPage()` skips `startViewTransition` in awakened mode.** The deck pages are hidden behind the fullscreen canvas in awakened mode — the page-transition animation is invisible AND it freezes the canvas. Awakened mode falls back to direct `renderPageBody()` calls. Deck mode keeps the smooth view transition (no canvas-freeze concern there because the dock canvas is a small embedded preview, not the main view). Per-emit deck-page re-renders still run unchanged so the Tab-overlay menu (`renderAwakenedPanelBody`) and the deck-mode pages stay live in the background; only the View Transition wrapper is bypassed for the awakened realm.
- **Result:** Periodic freezes eliminated. Awakened mode now runs at sustained high FPS without the ~250 ms hitches every 1-3 seconds. Deck-mode UX is unchanged — page transitions still cross-fade smoothly. The Tab-menu overlay in awakened mode shows live data exactly as before.

### Awakened combat + harvest polish — staff tip, reticle aim, magic-as-harvest, freeze (2026-04-19)

Single-session pass that took the awakened-mode combat / harvest loop from "the user can't reliably hit anything they're aiming at" to "everything works the way it visually appears to." Multiple tightly-related fixes are grouped here because they all chain off the same underlying bug class — the aim/origin/collision pipeline making different assumptions about Y in different stages.

#### 1. Bolt origin: real staff tip, not the wrist

- **Issue:** Magic orbs visibly leaked out of the avatar's WRIST, not the glowing finial of the wizard staff. At long range the wrong origin Y also tilted the bolt's flight vector enough to read as "magic landing below where I'm aiming."
- **Cause:** `getStaffTipWorldPosition()` returned `handR.position + (0, +0.4, 0)` — the wrist + a flat 40 cm bump. The actual finial sits ~0.98 m above the wrist (staff-local Y 1.103) and rotates with the staff's tilt + the hand's per-frame pose.
- **Solution:** Use `vanguardWizardStaffRoot.localToWorld(new THREE.Vector3(0, 1.103, 0))`. Magic number matches `orbCenterY = crownBaseY + 0.085` in `vanguardWizardLPCA.ts`. Falls back to legacy hand+offset when the staff root isn't available (boot races / non-vanguard variants).
- **Files:** `src/visual/characterScenePreview.ts` (`getStaffTipWorldPosition`).

#### 2. Reticle aim: mob-priority + fine terrain step + min-aim distance

- **Issue:** Free-aim bolts often landed "way below the reticle target." On uneven terrain the bolt would slam into dirt 4 m in front of the player even when the crosshair was clearly on a mob 22 m away. The "magic flies way too fast" complaint was the SAME bug — flight time at constant 14 m/s collapses from 1.6 s → 0.3 s when aim distance collapses from 22 → 4 m.
- **Cause:** `reticleAimPoint` walked the camera ray in 1.0 m terrain steps and picked `min(terrainHit, obstacleHit)`. A small terrain bump 4 m ahead produced a false ground crossing that beat the real mob hit at 22 m. No mob preference either, so creatures lost to terrain noise constantly.
- **Solution:** Three-candidate resolver in `reticleAimPoint`:
  1. **Mob hit** (Y-aware, `hitMobsOnly: true`) — promoted to top priority. Whatever creature the camera ray genuinely passes through wins aim regardless of terrain noise between you and them. Returns `mobOwnerId` so `castMagic` can soft-home onto the mob (free-aim bolts that grazed the cross-hair still hit a sidestepping target).
  2. **Static obstacle hit** (Y-aware, mobs filtered) — trees, walls, stations.
  3. **Terrain hit** with **0.4 m** step + binary refine to ~6 mm precision.
  
  Plus `MIN_AIM_DIST = 1.2 m` (static) / `0.4 m` (mob) so degenerate near-camera hits can't collapse aim to the player's feet.
- **Files:** `src/world/awakenedCombat.ts` (`reticleAimPoint`, `castMagic`, `AimResult` type, `resolveReticleAim` exposed on the handle).

#### 3. Camera floor-clamp broke camera-forward (zoom + elevation)

- **Issue:** Two seemingly-unrelated symptoms with one root cause. (a) **Zoom out misses the target:** zooming out drops the orbit camera Y enough to clip the terrain floor; clamp bumps Y up; reticle then aims at a different world ray than what the player thinks they crosshaired. (b) **Magic shot drifts left when high in a tree:** with the avatar at Y=5+, the camera's XZ floor sample was the ground BESIDE the tree (not the tree-top), so the floor clamp fired aggressively and tilted the camera while the right-shoulder offset stayed shifted right → muzzle-to-aim convergence went off-axis.
- **Cause:** `applyCameraFraming` bumped `camera.position.y` UP to clear terrain, but `lookAt` stayed at avatar height. Camera-forward = `(lookAt - position).normalize()` — when one endpoint moves, direction rotates. Standard third-person camera bug.
- **Solution:** When position.y is bumped by `deltaY`, apply the SAME `deltaY` to the lookAt target. Parallel translation of the eye frame — camera-forward direction is mathematically preserved. Same idiom we already use for the right-shoulder offset (shift both endpoints, not one).
- **Files:** `src/visual/characterScenePreview.ts` (`applyCameraFraming` — `lookAtBumpY` parallel-translate).

#### 4. Combat-ready avatar facing — staff visibly leaves toward reticle

- **Issue:** "Magic goes where character is looking, not where camera is pointing." When camera-locked but standing still (very common when sniping from a tree), the avatar held its last-walked yaw. Staff is on the avatar's right hand → if the avatar was facing west while the camera/reticle aimed south, the bolt visibly EXITED the staff heading north before line-straightening to the south aim point. Bolt LANDED at the right place but the visual exit direction was wrong, which reads as "shot fired in the wrong direction."
- **Solution:** Whenever `isCameraLocked()` is true, every frame snap-slerp `avatar.rotation.y` toward `atan2(cameraForward.x, cameraForward.z)` at 12 rad/s. Standard third-person shooter "combat-ready stance." When NOT locked, the legacy "facing follows movement velocity" path stays. The two paths can't fight: the camera-locked branch runs first and the velocity path is gated `if (!isCameraLocked)`.
- **Files:** `src/world/freeRoamControls.ts` (per-frame `update` — combat-ready facing block + skip velocity-facing when locked).

#### 5. Y-aware bolt collision raycast — bolts no longer despawn on trees they fly OVER

- **Issue:** "Magic doesn't reach the reticle when shooting from a tree." With the avatar at Y=5 firing at a target 25 m away, the bolt detonated mid-air a few metres in front of the player — even though the visual line should have been clear. Same thing for shooting over your own cabin walls or between rocks.
- **Cause:** The per-frame collision check used `collisionWorld.raycastXZ` in 2D-only mode (no `originY` / `dirY`). That treats every blocking footprint as INFINITELY TALL. A bolt at Y=9 flying over a tree topping at Y=8 still "hit" the tree's flat XZ footprint and despawned.
- **Solution:** Pass `originY: prevY` and `dirY: segDy / segLen` to the raycast (the Y-aware mode added in the 2026-04-19 stations + magic pass). The bolt's actual Y at the candidate footprint's XZ is checked against the footprint's `[bottomY, topY]` band — a bolt above the canopy sails clean past.
- **Files:** `src/world/magicProjectiles.ts` (per-frame collision block — `prevY` capture + `dirY` plumb-through; uses the existing Y-aware raycast contract from `collisionWorld.ts`).

#### 6. Magic-as-universal-damage / magic-as-harvest

- **Issue:** Bolts only damaged mobs. Hitting a tree / rock / shrub was visually a wasted shot. User wanted: "shots from magic need to do damage to building, enemies, character, trees, rocks, everything — and just as when we harvest, that object has an animation as it despawns; from a distance the player gets the resource added to their inventory."
- **Solution:** Generic `onStaticHit(ownerId, hx, hy, hz, damage)` callback on `MagicProjectilesHandle`. The bolt's existing collision handler dispatches non-mob hits to the callback (mob hits keep their existing path). `mountApp.ts` parses the owner-id prefix and routes:
  - `harvest:<kind>:<index>` → `harvestHandle.applyHit(node, store.getHarvestHitsMultiplier(kind))` — the SAME pipeline as a melee press, with the player's currently-equipped tool multiplier so yield + per-hit chip + final-yield + tool-wear all match exactly. Despawn animation (tree-fall / rock-crumble / etc.) is the same too because it's owned by the harvest module.
  - `dock-forest-batched:<kind>:<idx>` → `dockForestHandle.getNodeByIndex(idx)` then same `applyHit` flow with `yieldKindForObsKind` mapping.
  - `cabin:` / `craft_station:` → bolt despawns silently (no HP system on player builds yet).
  
  Added `getNodeByIndex(idx)` on `DockForestHandle` so the routing avoids scanning the full nodes array. Cyan damage floater spawns at the hit point so the player gets `-N` chip-yield feedback at distance.
- **Files:** `src/world/magicProjectiles.ts` (`onStaticHit` opt + dispatch in collision block), `src/world/dockForestBatchedScene.ts` (`getNodeByIndex` + `collisionRadius` on the public node type), `src/world/freeRoamHarvestNodes.ts` (`collisionRadius` on `ScatteredNode`), `src/ui/mountApp.ts` (callback wiring with full ownerId-prefix routing).

#### 7. Reticle-only harvest dispatch (when camera-locked)

- **Issue:** Standing in the creek next to a tree, pressing E always filled the bucket (water won by proximity) — never harvested the tree the reticle was on. Also: with no reticle priority, gathering was unpredictable in mixed-resource clusters; the player would chop a fern when they wanted the rock right next to it.
- **Solution:** In awakened + camera-locked mode, the reticle is the SOLE target picker. Two gates BOTH have to pass:
  1. **Reticle gate** — the camera-ray's world hit point lands on a harvestable thing.
  2. **Physical-reach gate** — the avatar is within `HARVEST_AVATAR_REACH = 1.3 m` of the resource's collision SURFACE (centerDist - collisionRadius), so a giant oak with a 1.5 m trunk radius is reachable when the player is touching the bark, not unreachable forever as it would be with a center-distance check.
  
  If either gate fails the press is consumed silently — NO proximity fallback. "Press E" prompts also hidden in awakened mode (universal-harvest contract is "anything visible is harvestable").
  Free-cursor mode (Q off) keeps the legacy proximity dispatch for non-combat dock interaction.
- **Files:** `src/ui/mountApp.ts` (`onInteract` reticle-pick block + prompt hide), `src/world/awakenedCombat.ts` (`resolveReticleAim` exposed on handle for reuse).

#### 8. Tree collision shrunk to TRUNK — big trees harvestable

- **Issue:** "Lots of trees won't let me harvest, won't let me get close enough." Max-scale dock-forest oaks (`radius * sm` reaching 1.6 m+) pushed the player out at center-distance > the legacy 1.8 m harvest gate. No matter how close you stood, the gate failed.
- **Solution:** Two independent fixes that compound:
  - Shrink per-species collision radii to match the actual TRUNK silhouette (was canopy-edge): apple 0.40 → 0.24, pine 0.32 → 0.20, birch 0.28 → 0.18, fir 0.34 → 0.22, oak 0.42 → 0.26. Awakened scatter `apple_tree` 0.45 → 0.30. Player can now walk *under* the canopy and stand right next to the bark. Visible canopy stays as decoration; nothing actually requires landing on top of a tree.
  - Use SURFACE distance instead of CENTER distance in the harvest-reach gate (`centerDist - collisionRadius <= HARVEST_AVATAR_REACH`). Required exposing `collisionRadius` on `ScatteredNode` and `DockForestHarvestNode` public types.
- **Files:** `src/visual/forestEnvironment.ts` (`TREE_SPECIES_RADIUS_PER_SM`), `src/world/freeRoamHarvestNodes.ts` (`HARVEST_FOOTPRINT_RADII.apple_tree` + `collisionRadius` on `ScatteredNode`), `src/world/dockForestBatchedScene.ts` (`collisionRadius` on `DockForestHarvestNode`), `src/ui/mountApp.ts` (surface-distance gate).

#### 9. Combat freeze — debounced `saveState`

- **Issue:** "When animals attack me and I attack them, aspects freeze." Visible stutter every hit during combat.
- **Cause:** `saveState()` is called from 23 mutating store methods. Each is a synchronous `JSON.stringify(state)` + `localStorage.setItem` of a multi-KB object — 15-50 ms of main-thread block per call. Awakened combat fires saves at high frequency: useMana on each cast, damageAwakenedMob on each bolt hit, damagePlayerInAwakened on each mob bite. With 6 mobs alive that's 7-10 saves per second of a multi-KB state — visible stutter on every event.
- **Solution:** Coalesce `saveState` calls within a `SAVE_DEBOUNCE_MS = 250 ms` window into a single deferred write. The latest snapshot reference always wins (one write of the final state instead of 20 sequential writes of intermediates). `pagehide` / `beforeunload` / `visibilitychange` flush the pending save synchronously so a normal browser close still persists everything. Worst-case crash loss is 250 ms of state changes — invisible for combat ticks, irrelevant for purchases (subsequent mutations refresh the timer anyway).
- **Files:** `src/core/gameStore.ts` (`saveState` debounce + `flushPendingSave` + `pagehide`/`beforeunload`/`visibilitychange` listeners).

#### 10. Picker modal z-index — spell + consumable pickers float above inventory

- **Issue:** Click an empty spell or consumable slot from the inventory page (Tab → Inventory) and the picker rendered UNDER the inventory overlay. Player had to close inventory first to interact with the picker.
- **Cause:** `spellPickerModal` and `consumableSlotPickerModal` used `position: absolute` with `z-index: 95`. The picker mounted as a child of `shell`, but the awakened menu overlay (`.awakened-menu-overlay`) is `position: fixed; z-index: 9100` — that creates a higher stacking context the absolute picker can never escape.
- **Solution:** Change both pickers to `position: fixed; z-index: 9300`. Now they participate in the global stacking context and float above any open page overlay.
- **Files:** `src/ui/spellPickerModal.ts`, `src/ui/consumableSlotPickerModal.ts`.

**Cross-cutting "do NOT" list for this pass:**

- **Don't** read `camera.getWorldDirection()` after mutating `camera.position` if you haven't called `lookAt()` again — the matrix is stale.
- **Don't** clamp `camera.position.y` without applying the SAME bump to `lookAt` — you'll silently rotate the camera-forward vector and break aim.
- **Don't** use 2D-only `raycastXZ` for any swept-3D query (projectiles, line-of-sight, etc.). The 2D mode treats footprints as infinitely tall and produces ghost hits on anything the ray flies above. Always pass `originY` + `dirY`.
- **Don't** call `saveState` synchronously in any high-frequency hot path. Use the debounced version exclusively from now on; if you need a guaranteed flush (death, page unload), call `flushPendingSave()` directly.
- **Don't** mount picker modals with `position: absolute` if they need to float over `position: fixed` overlays — different stacking contexts. Always `position: fixed` for true overlay modals.

### Ground-level lighting overhaul — IBL, camera fill, half-Lambert, night grade, god-rays (2026-04-19)

Eight-phase research-driven lighting pass implementing every recommendation from `docs/GROUND_LEVEL_LIGHTING_OVERHAUL_PLAN.md`. Inspired by Echoes of Wisdom / BotW / Genshin recipes + the sibling GoE project's `LIGHTING_SYSTEM.md`, `DAY_NIGHT_CYCLE_SYSTEM.md`, and `SKY_REFLECTION_FOG_RESEARCH.md` (`C:\GameofEmpiresDocs\docs\`). All eight phases ship together; each is a small composable layer that can be tuned or rolled back independently.

#### 1. PMREM IBL from a procedural sky-gradient sphere

- **Issue:** `scene.environment` was null. Many `MeshPhysicalMaterial`s carried `envMapIntensity` values (0.5–1.1) but had no env map → the entire environment-reflection term was dead. PBR materials read "plasticky" or "muddy" depending on view angle.
- **Solution:** New `IdleCraftDockEnvironment.attachIbl(renderer)` builds a tiny env scene with a single inside-out gradient sphere (3-color shader: zenith / horizon / nadir + sun-glow lobe). PMREM regenerates the prefiltered env target every ~4 s; `scene.environment` swaps to the new texture and the previous target is disposed AFTER swap. Per-frame uniform sync uses the same sky-tone state the dock environment already computes for fog (`dayMix`-blended zenith, sunWarmth-tinted horizon, sun direction for the glow lobe).
- **Why a gradient sphere instead of cloning the actual skydome:** the actual skydome has a multi-band shader + cloud occlusion via `renderOrder`. Cloning would risk material-state desync. The 3-color gradient + sun-glow captures the ~5 most important sky directions which is more than enough for IBL specular — IBL's job is "what's the dominant color in each direction" not "render the sky exactly."
- **Files:** `src/world/idleCraftDockEnvironment.ts` (`attachIbl`, `updateIbl`, IBL state + dispose), `src/visual/characterScenePreview.ts` (calls `attachIbl(this.renderer)` after dock-env binding).

#### 2. Camera-relative fill light (third-person "softbox")

- **Issue:** With sun + hemi + ambient only, the avatar reads silhouette-flat at noon when their back is to the sun. Industry-standard recipe (BotW / Genshin / Tunic) attaches a low-intensity directional to the camera rig itself.
- **Solution:** Phantom `DirectionalLight` parented to `this.camera` at attach time. Position `(1.4, 0.6, 0)` = right-shoulder over-the-eye-line, target forward. `castShadow = false` so it doesn't add to the shadow-light count. Per-frame intensity + color driven by the dock environment via `bindCameraFillLight(light)`: intensity `0.20 × dayMix + 0.10 × nightMix × moonStrength`, color lerped from cool → warm with sun warmth, then toward moon-base when nightMix dominates. Color always tracks the sky so it never reads as a "fake studio light."
- **Files:** `src/visual/characterScenePreview.ts` (allocate `cameraFillLight`, parent to camera, call `bindCameraFillLight`), `src/world/idleCraftDockEnvironment.ts` (`cameraFillLight` field, `bindCameraFillLight`, per-frame drive in `update`).

#### 3. Half-Lambert wrap on hero materials

- **Issue:** Standard PBR `max(0, dot(N, L))` produces near-black shadow sides on stylized characters/mobs at noon. Industry recipe: the half-Lambert wrap (Valve Source 2004 → BotW → every modern stylized title).
- **Solution:** New `installHalfLambertOnMaterial(mat)` and `installHalfLambertOnTree(root)` in `src/visual/halfLambertLighting.ts`. Patches the lighting model via `onBeforeCompile` + `customProgramCacheKey` (same WeakSet-deduped pattern as `installVegetationWindOnMaterial` — proven safe). Replaces the canonical `vec3 irradiance = dotNL * directLight.color;` line with `(pow(dotNL * 0.5 + 0.5, 2.0)) * directLight.color`. Specular path's `geometry.dotNL` untouched, so metals still read.
- **Applied to:** every player-avatar material (skin, undertunic, jerkin, trim, pants, boots, hair, hat band/top/brim, artisan hair) + every PVE enemy (rat, wolf, raider/wanderer). Terrain, fog, water deliberately NOT patched.
- **Files:** `src/visual/halfLambertLighting.ts` (new), `src/visual/characterScenePreview.ts` (loop over `avatarSkinMat ... artisanHairStreakMat`), `src/visual/pveEnemyLPCA.ts` (`installHalfLambertOnTree(root)` after build, before grounding).

#### 4. Night-grade post-process pass (moonlight reads as moonlight)

- **Issue:** Just dimming the moon makes night look like "blue daytime." The brain reads "night" because of frame-wide desaturation + cool tint + gamma curve, not because the dominant light is dim.
- **Solution:** New `src/engine/nightGradePass.ts` — a `ShaderPass` that desaturates 45 %, cool-tints `(0.85, 0.92, 1.08)`, and crushes mid-shadows with `pow(rgb, 1.25)`. Strength scales with `u_nightMix`; full-moon (`u_moonIllum`) eases the grade by 30 % so full-moon nights stay readable while new-moon nights are the "mysterious dark" extreme. Pass is `setPassEnabled(false)` at noon (program stays in cache — per LEARNINGS post-stack invariant — but doesn't run). Injected into the composer via `EffectComposer.insertPass` BEFORE the vignette pass so vignette darkens the already-graded image.
- **Files:** `src/engine/nightGradePass.ts` (new), `src/visual/characterScenePreview.ts` (`nightGradePass` field, `ensureNightGradePass()` injection inside `applyDockPostProcessing`, per-frame `syncNightGradeUniforms` after dock env update), `src/world/idleCraftDockEnvironment.ts` (`getNightMix()`, `getMoonIllum()` getters).

#### 5. Horizon-matched fog colors (atmospheric perspective)

- **Issue:** Fog color was lerping toward zenith blue. Distant objects matched the sky-top but felt disconnected from the warm sunset / cool moonlit horizon bands. No atmospheric perspective → character pop suffered.
- **Solution:** Two new private colors (`horizonWarm`, `horizonCool`) on `IdleCraftDockEnvironment`. New `_tmpHorizonFog` scratch. After the base sky lerp, compute `horizonInfluence = 1 - smoothstep(sun.y, 0.0, 0.45)` so the warm/cool horizon tint only kicks in when the sun is near the horizon (dusk/dawn/night). Color lerps `horizonCool → horizonWarm` by `sunWarmth`. At noon: tint ≈ 0, sky-zenith match preserved. Tightened night fog band (`near × 0.42`, `far × 0.55`, was `× 0.28 / 0.35`) so distant objects desaturate harder at night.
- **Files:** `src/world/idleCraftDockEnvironment.ts` (`horizonWarm` / `horizonCool` / `_tmpHorizonFog` fields, fog block rewrite).

#### 6. Cheap cone-geometry god-rays

- **Issue:** Cinematic moments (sun through canopy, magic in dust) felt ordinary. GoE's `SKY_REFLECTION_FOG_RESEARCH.md` calls out this trick as "perfect for stylized look, fits stylized fantasy" — additive cone geometry mesh oriented along sun direction.
- **Solution:** New `src/visual/sunGodRays.ts` exports `attachSunGodRays(scene)` returning a handle. Builds an open `ConeGeometry(18, 60, 24, 1, true)` translated so apex sits at world origin, base extends upward. `MeshBasicMaterial` (additive, depth-write off, transparent, double-sided). Per-frame `update({sunDir, sunDirectFrac, sunWarmth, storm})` sets quaternion via `setFromUnitVectors(up, sunDir)` and modulates opacity (`sunDirectFrac × 0.06 × (1 - storm × 0.6)`) + color (lerp neutral → peach by sunWarmth). Cost is essentially free — additive transparent program is one of the most common Three.js shaders, already cached.
- **Files:** `src/visual/sunGodRays.ts` (new), `src/world/idleCraftDockEnvironment.ts` (`godRays` field, `attachGodRays()`, per-frame update after `sunDirectFrac` is computed, dispose in cleanup), `src/visual/characterScenePreview.ts` (calls `attachGodRays()` after `attachIbl`).

#### 7. Eye-adaptation lite (asymmetric exposure smoothing)

- **Issue:** `getExposureMultiplier` returned the raw target each frame. Dawn/dusk had perceptible exposure steps when `dayMix` swept through its smoothstep band.
- **Solution:** Replaced with smoothed-toward-target `adaptedExposure`. `tau = 0.3 s` when going brighter (eyes pinch fast), `tau = 1.2 s` when going darker (eyes adapt slowly to shadow). First-tick snap to target so boot doesn't look like a slow lerp from 1.0. Removed obsolete `lastExposureDayMix` / `lastExposureStorm` state fields the old direct path was caching — they're never read now.
- **Files:** `src/world/idleCraftDockEnvironment.ts` (`adaptedExposure` field, eye-adaptation tick block in `update`, `getExposureMultiplier` returns smoothed value).

#### 8. Bloom tuning (emissive props pop, mundane surfaces don't)

- **Issue:** `project.json` `bloomThreshold: 0.05` made everything brighter than 5% bloom — noon grass blooms, white clouds bloom, bright cabin wood blooms. The magic projectile orb (which IS the thing that should look magical) got lost in the general glow.
- **Solution:** `project.json`: `bloomThreshold` 0.05 → 0.85, `bloomStrength` 0.04 → 0.45, `bloomRadius` 0.04 → 0.4. Only HDR-bright surfaces bloom now. Bumped magic-projectile emissive intensity (midShell 0.9 → 1.6, crystal facets 1.6 → 2.4) so they reliably exceed the new threshold and visually win the bloom pass — emissive props read as "magic" rather than "everything is glowing."
- **Files:** `project.json` (postProcessing tuning), `src/world/magicProjectiles.ts` (emissiveIntensity bumps with comment referencing the bloom-threshold change).

**Cross-cutting LEARNINGS-invariant compliance (all 8 phases):**

- `numDirLightShadows` constant — Phase 2 fill light has `castShadow = false`; doesn't add to shadow count.
- `numPointLights` constant — no new point lights in this pass.
- `setPassEnabled` not dispose — Phase 4 night grade uses `pass.enabled = nightMix > 0.05` toggle; program stays in cache.
- `onBeforeCompile` + WeakSet + `customProgramCacheKey` — Phase 3 half-Lambert mirrors the proven vegetation-wind pattern exactly.
- First-use shader compile freeze — Phase 4 pass is allocated at composer-build time so the program compiles in the boot warm window. Phase 1 PMREM does its initial compile during `attachIbl()` which is called in the dock attach sequence.
- All eight phases gracefully no-op when their feature is disabled (post stack off, no renderer, etc.).
